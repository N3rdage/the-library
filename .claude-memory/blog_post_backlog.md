## Strong catch-up candidates

Concrete angle, durable lesson, transferable beyond BookTracker's stack. Order is rough — pick whichever speaks to current energy.

- **"0 warnings was a lie my build told me"** — same arc. Reported the motion PR warning-clean off `dotnet build`; Drew ran a full rebuild and surfaced 5 `CS0618`s in *my own* code (net10 MAUI obsolete-API). Incremental builds only recompile changed assemblies, so warnings in untouched code — and in just-edited files once their assembly caches — don't reappear: "0 warnings" off an incremental build is meaningless. Transferable to any incremental-compilation toolchain; pairs with the warnings-as-errors tech-debt (TD-11) and the verify-don't-assume-green thread (#6/#7/#12). Anchor: [[retro_bookshelf_redesign_arc]] + [[feedback_verify_warnings_clean_build]].

- **"When the documented fix isn't enough" — the slot-swap wedge recovery ladder** — 2026-06-11 prod incident. A slot swap left both App Service slots reporting `Running` with the DB `Online`, but neither served traffic — and this time the *documented* recovery (SQL DB restart, the fix locked in by the 2026-05-08 incident) wasn't enough. Symptom escalated: 504s, dead-silent Log Stream, zero App Insights, and eventually SCM itself 504'd — the wedge was below the app layer, at the instance level. Resolution required climbing the ladder all the way to an App Service Plan SKU migration (legacy S1 → Premium V3, conveniently cheaper + higher spec) to force the workers onto fresh compute. The durable lesson: runbooks encode the fix that worked *last* time, but a recovery *ladder* (slot Stop+Start → SQL restart → plan compute migration) survives the case where last time's fix stops working; and the tell for "go straight to the top of the ladder" is when the management surfaces (SCM / Log Stream) are themselves unresponsive. Audience: anyone running Azure App Service + slot swaps; ops-minded devs. Companion to the deploy-as-incident postscript in shipped post #8. Anchor: [[runbook_sqlclient_pool_zombies]] (second confirmed incident) + this session.

- **"Filters are URLs"** — the Library nav rework (2026-06-12, PRs #322/#323). The Library page reset every filter whenever you drilled into a book and pressed Back, because the view-model was `Transient` — a fresh instance with default filters on every navigation. The fix wasn't more state management; it was *less* — move the whole filter set into the query string via `[SupplyParameterFromQuery]` so a filtered view *is* a URL. The browser's own Back button then does the work (filters restored, bookmarkable, shareable), and group rows become drill-down links to filtered flat lists rather than inline-expanding accordions. The transferable lesson: when per-navigation state keeps getting lost, the URL is often the right store — it's the one piece of state the platform already persists and restores for you. Blazor-Server-specific in the details (Transient VM, `OnParametersSetAsync` hydration, `replace:true` to keep history clean) but the principle generalises to any SPA. Risk: needs a tight through-line so it doesn't read as a Blazor tutorial. Anchor: this session's PRs + the `BuildGroupDrillParameters` extraction (which is also where post #12's bug lived). Note the natural pairing — #12 ("Three reviewers") is the *review* story from this same arc; this is the *design* story.

- **"Three Dependabot PRs, three resolution shapes"** — 2026-05-12 batch resolution. Three failing major-version Dependabot PRs on BookTracker (bunit 1→2, Respawn 6→7, Microsoft.ApplicationInsights.AspNetCore 2→3) demonstrated three distinct categories of "major version bump" with three distinct responses: (a) **mechanical renames** (bunit: `TestContext` → `BunitContext`, `RenderComponent` → `Render`) — migrate in place, minutes; (b) **API surface changes** (Respawn: `string connectionString` → `DbConnection`) — refactor call sites, minutes; (c) **architectural rewrites** (AppInsights AspNetCore 3.x is a full OpenTelemetry replacement of the classic SDK; `ITelemetryInitializer` doesn't exist) — NOT a routine bump, defer with a `dependabot.yml` ignore-by-update-types rule and a captured-scope TODO. Plus the bunit subplot — compile passed after the mechanical rename, but the 9 component tests failed at runtime because MudBlazor's services register as `IAsyncDisposable`-only and bUnit 2.x's sync `Dispose` couldn't handle them; fix was opting the test base into xUnit's `IAsyncLifetime`. Underused-but-valuable insight: `update-types: ["version-update:semver-major"]` in `dependabot.yml` lets a project auto-apply minor/patch security/CVE patches while holding major bumps for human review. Audience: solo devs + small teams running Dependabot. Transferable lesson — major bumps aren't a homogeneous category; categorise the resolution shape before touching code.

- **"Author-mix breaks per-author rules"** — 2026-05-22 Destroyer-series genre converge. The 2026-05-18 `backfill-author-default-genres.sql` shipped an elegant per-author default-genre mapping (Sapir → Thriller+Adventure, Murphy → Thriller+Adventure, etc.) gated on "Work has zero existing genres". The mapping read cleanly in isolation. In practice, the Destroyer series — co-authored across Sapir + Murphy + various ghostwriters, with capture history mixing manual tags + Open Library hits + earlier scripts — fragmented into a noise pattern: 35 Works carrying 65 GenreWork rows split across Adventure (35), Thriller (26), Mystery (4). The per-author rule never had a stable answer for "what genre is *this* Destroyer book" because authorship varied per row. Audience: data-modelling devs, anyone backfilling categorical data from per-row attributes that aren't actually per-row. - **"What it takes to keep an AV scanner happy"** — the encoded-PowerShell-args journey, anchored by `feedback_avoid_encoded_powershell.md` and a sequence of in-session AV trips spanning the project: (a) original commit-message here-string trigger; (b) BuildInfo arc multi-statement smoke checks (`dotnet build ...; $dll = ...; if (Test-Path $dll) { ... }`); (c) 2026-05-08 multi-work-collections PR2 API investigation, where even a single-statement `Invoke-WebRequest` with a long URL got encoded — and individual short calls in a parallel batch *also* tripped. The post traces the workarounds in priority order (temp file in `$env:TEMP` not `.git/`; short inline `-m`; bash heredoc that doesn't work for git on Windows; final-fallback "user runs the script in their own terminal so the encoding wrapper isn't in the path") and names the security-aware framing that emerged from a side conversation: when a harness will inevitably write somewhere outside its session-local memory, *where* matters. `.git/` is a privilege-escalation surface (hooks are RCE-equivalent on the user's next git op; config write hijacks `remote.origin.url`); the OS temp dir is the right answer because it's outside the repo and has no security sensitivity. Drew's framing of "this is a great post — Claude-vs-AV is real and ongoing" gives it the narrative arc. The deeper lesson is "fix the workflow, not the security tool" — the AV is correctly flagging an indistinguishable pattern; the answer isn't to whitelist (compromising the user's defence-in-depth), it's to stop generating encoded commands in the first place. Audience: developers running LLM agents on Windows, security-conscious devs in any AI-collab setting, broader audience for "false positives that are actually true positives for the right reason." Drew flagged the topic in-chat (2026-05-08) as blog material ("Need to write a blog post of Claude vs AV Software, it will be amazing"); add the .git/-vs-temp-dir conversation as the "did the right thing" beat that pulls the post above pure war-story.

## Going-forward pattern

After this catch-up corpus is mined, new retros add their own backlog entry at write time:

- When a retro is written, add a one-bullet candidate post at the top of "Strong" or "Possible" with the source retro link.
- When a post ships, move its entry from candidates to "Already shipped."
- When the catch-up "Strong" list empties, the going-forward pattern is the only source of new candidates — no more sweeps needed.

## Suggested starter prompt for picking the next post

In a fresh session, this is enough to choose the next one to draft:

> Let's pick the next blog post from the backlog. Suggest 3 from the strong-candidate list with one-line angles and which retro is the anchor.
