## Shipped

PR #95 — new `/books/{id}` route with a read-only MudBlazor detail page. Single-Work books (the common case) collapse Book and Work into one "Details" panel; multi-Work compendiums (Complete Works of Shakespeare, Lovecraft omnibus) render a filter-searchable expandable list of rows. Editions + copies render as MudExpansionPanels with a summary line ("N editions · M copies"). Tags display as chips. Four rollout decisions in PR 1:
1. Library list click target unchanged — still goes to `/edit`. The new View is reachable only via a temporary "View (preview)" link in the Edit header so the rest of the app doesn't commit to it yet.
2. No inline edits, no modals — pure browsing. Edit surfaces land in PR 2–4.
3. MudBlazor continuation — the warm library theme applies automatically.
4. Compendium scale handled lightly — compact rows + filter input when Works > 10. No virtualize; 37 Shakespeare plays is 37 DOM rows, well within budget. Revisit if scroll perf becomes a real issue.

## Surprise

- **The PR-1 scaffold shipped cleanly, but the merge almost didn't.** Drew said "merged it", I pulled — remote main was still at the previous PR. Turned out he'd forgotten to hit Enter on the GitHub merge button. Caught it by `git ls-remote` showing `origin/feat/book-view-page-pr1` still alive at my commit SHA, which it wouldn't be if the branch had been auto-deleted on merge. Cost: one "hmm, state seems off" round-trip before proceeding. The lesson here isn't "Drew made a mistake" — it's "before acting on a 'merged' handoff, verify with the actual remote state, not just the user's message." The cost of checking (`git fetch && git log origin/main`) is zero; the cost of building on top of a phantom merge is real.
- **`MudList` and `MudExpansionPanel` in MudBlazor 9 have small shape surprises.** `MudList` requires a `T=` type parameter on both the list and every item (I used `T="int"`); `MudChip` similarly wants `T="string"` in 9.x. These weren't obvious from the quickstart docs and produced compile errors rather than runtime weirdness, so cheap to fix but jarring if you came in expecting the old untyped API. First-PR friction worth noting for future MudBlazor work.
- **Single-Work vs multi-Work as a data-driven UX collapse actually feels right on paper.** I was half-expecting to ship it and realise "oh, the mental model leaks somewhere" — e.g. where does "first published" live for a multi-Work book? It's per-Work, so the compendium view shows it inside each expanded row; the Book itself doesn't have one. That sorted out naturally. The Work refactor (which lifted subtitle/author/genres/series off Book onto Work) turned out to be exactly the shape the View page needed; if those fields had still been split across both entities the UI decision would have been much harder.
- **The test assertion I wrote was arithmetically wrong, and the test caught it.** Tests passed on three cases, failed on "Editions + Copies count + nest" with the helpful diff "Expected 2, Actual 3". The bug was in the assertion (`e0.Copies.Count + e1.Copies.Count` compared to `2`, should've been `3`, or better: asserted per-edition counts separately). Classic case of "test says what I meant to say, not what I wrote." Fix was trivial. Worth noting because it's a small argument for writing the assertion as the domain statement ("one edition has 2 copies, the other has 1") rather than the arithmetic sum — harder to get the domain wrong than the arithmetic.

## Lesson

- **Ship the scaffold PR small and make the rollout opt-in.** The View page is reachable only via a "View (preview)" link on Edit. The Library still routes to Edit. That sounds timid, but it's the right call: PR 1 is the UX bet ("does the mental model work?") and should be evaluatable without committing the whole app to it. If the mental model is wrong, PR 1 reverts cheaply; if it's right, PR 2 swaps the Library link. Future UI experiments should default to this shape — introduce the new surface without rerouting, let the user click both surfaces side-by-side, then commit.
- **PR scaffold = ViewModel shape + page skeleton + one test per non-trivial projection.** No inline edits, no modals, no business logic. Everything beyond that gets deferred to named PRs in the plan (PR 2 inline saves, PR 3 modals, PR 4 add flows, PR 5 browse flows). This keeps PR 1 tightly reviewable and gives future-me a natural seam — each subsequent PR has a clear "delta from last" instead of being a Christmas tree of half-started ideas.
- **`IsReadOnly` view models are simpler than they look.** `BookDetailViewModel` is under 150 lines: load, project into records, expose three computed properties. No form-input shapes, no dirty tracking, no SaveAsync. Compare to `BookEditViewModel` at ~550 lines. Read-only VMs should feel small — if they don't, you're probably eroding the separation. The discipline: the display shape on the VM is what the Razor template consumes, nothing more. If the template needs a computation, it goes on the VM; if the template needs to mutate, it goes to a different VM (or a modal with its own VM) in a later PR.
- **"Convert as we touch" keeps paying off.** This is the second MudBlazor-only page (after Home + MergeBook in the pilot PR). The palette applied automatically; the component-library namespace doesn't collide with anything Bootstrap touches; Detail.razor doesn't need to know that Edit.razor down the hall is still Bootstrap. The only piece of "coexistence" this PR touched was adding a Bootstrap-styled link on the Edit page pointing at the MudBlazor page — which worked without ceremony because links don't care about component libraries.
- **Before Reading a PR is done, check the remote.** See Surprise #1. Added to the mental protocol: when a merge-confirmation arrives, first step is `git fetch && git log -1 origin/main` — if the expected SHA isn't there, flag it before doing any post-merge work. Cheaper than unwinding phantom state.

## Quotable

"Drew said merged, but the remote says otherwise." The most useful thing I can do when a state-changing action is reported — merge complete, deploy complete, CI passed — is verify the actual artefact before building on it. The user's message is a claim; the remote is ground truth. The same logic applies to `az webapp auth show` vs the REST API, to `kubectl get` vs cluster state, to the CI green-check vs the artefact hash. Every system that reports state has a slower-than-real-time view of that state. The fix is boring and universal: read the source of truth before you act on the report.
