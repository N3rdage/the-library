## Shipped

PR #97 — inline auto-save for four fields on `/books/{id}`:
- **Rating** (`MudRating`): click-to-save, silent.
- **Status** (`MudChipSet<BookStatus>` with `SelectionMode.SingleSelection`): three filter-chip buttons (Unread / Reading / Read), tap-to-save, silent.
- **Notes** (`MudTextField` multiline inside a new `NotesEditor` child component): 800ms debounced save after last keystroke, immediate flush on blur. Save-state indicator ("— saved" / "— saving…" / "— unsaved") sits next to the label so the user can always see whether their typing reached the DB.
- **Tags** (`MudAutocomplete<string>` + closeable `MudChip`s): autocomplete against existing tags (excludes already-assigned). Enter or blur commits. Case-insensitive normalisation reuses the existing Tag row rather than duplicating. One-click remove.

Errors surface via `ISnackbar`. Structural edits (Work, Edition, Copy) remain read-only — those land in modal dialogs in PR 3.

## Surprise

- **Silent saves are fine for single-value fields; textareas need a visible state indicator.** When rating / status save, the UI immediately reflects the new value (a new star, a different chip filled in), so a "Saved" toast would be redundant. Notes are different: typing into a textarea is *ambiguous* — did my last few keystrokes make it to the DB or did the circuit break mid-save? The "— saved" / "— saving…" / "— unsaved" marker beside the label is the smallest acceptable UX for a debounced textarea. Without it, users either (a) pause anxiously to watch for *something* happening, or (b) assume nothing saved and re-type. The rule generalises: inline auto-save for fields where the value IS the feedback → silent. Inline auto-save for fields where the value is *content the user wrote* → visible save state.
- **`CancellationTokenSource` + `IAsyncDisposable` was cleaner than a `Timer` for the debounce.** First instinct was `System.Timers.Timer` or `PeriodicTimer`, then I remembered Blazor Server components are disposable and owning a mutable Timer across async boundaries is awkward. The CTS pattern is ~10 lines: each keystroke cancels the previous token and starts a new delayed save. `DisposeAsync` cancels any pending save so a torn-down circuit doesn't fire a DB write into the void. The pattern is basically "the 'latest-wins' debounce is a one-shot Task per keystroke; the cancellation token is the whole lifecycle." Simpler than I'd expected, will reuse.
- **`MudChipSet` with `SelectionMode.SingleSelection` is the best status widget for mobile.** I considered `MudSelect` (classic dropdown), `MudRadioGroup` (big touch targets but chunky), a segmented `MudToggleGroup` (MudBlazor 9 has it but the styling is fiddly). The three-chip ChipSet wins: all three states visible at once, current state filled + coloured (Default / Info / Success), one tap to switch, renders tight on mobile. Worth remembering for any small enum with 2–4 values where "show all options + highlight current" beats "show current + expand to pick".
- **The `NotesEditor` child component sat right on the edge of "overkill" and earned its keep.** Mild temptation to inline `MudTextField` + indicator into the two panels (single-Work Details + multi-Work "Works in this book"). Resisting saved me ~20 lines of duplication and centralised the dirty/saving/saved marker. The split is *specifically* worth it when the same editor appears in two code paths with no conceptual divergence. If the two notes fields ever needed different behaviour (e.g. per-work notes vs book notes), I'd flatten the child component back out.
- **Case-insensitive tag find-or-create worked first try because the existing `BookEditViewModel` had the same pattern.** `name.Trim().ToLowerInvariant()` → `db.Tags.FirstOrDefaultAsync(t => t.Name == normalized)` → reuse or create. The Add page and Edit page already normalise tags this way, so all three entry points converge on the same canonical row. Would have needed a migration to dedup existing mixed-case duplicates if we were introducing this from scratch, but the convention was already established. Consistency-in-codebase paying forward again.
- **Dev-server DLL lock caught us again.** Pattern #4 in `patterns.md` (Browser-test honesty) already documents that the running dev server holds `BookTracker.Data.dll` and blocks `dotnet build`. Hit it today — build failed with `MSB3021` file-lock errors, Drew stopped the server, build ran clean in 7 seconds. Not a new lesson but a useful confirmation that the documented workflow ("ask Drew to stop the dev server briefly so I can verify the build") is still the right protocol. The right answer isn't "invent a workaround"; it's "the documented friction is correct, use it."

## Lesson

- **Inline auto-save belongs to the fields where the value *is* the feedback, plus one exception.** Rating, status, single selection, toggle switches — silent saves because the UI already shows "this is the new state". Text content (notes, description, long-form) — visible save state because the textarea is mutation-not-selection and users need to know if their content reached the backend. The exception-to-silent is worth carving out explicitly: **if the user had to *compose* the new value, the save needs a visible confirmation.** Applies beyond BookTracker — this is a durable UX rule.
- **Debounce = CTS + IAsyncDisposable. Don't reach for Timer.** In a disposable component (Blazor Server circuit, React effect with cleanup, any async-capable UI runtime), the debounced save is:
  1. Cancel the previous token.
  2. Dispose it, create a new one.
  3. `_ = DelayedSaveAsync(token)` — fire-and-track, not awaited from the event handler.
  4. On dispose, cancel + dispose the latest token.
  This is ~15 lines, has no long-lived state beyond the single CTS field, and behaves correctly under rapid keystrokes + abrupt disposal. Simpler than Timer, simpler than Rx, works in any async-capable UI.
- **Stop re-inventing widgets; the library already has the right primitive.** `MudChipSet` with `SelectionMode.SingleSelection` is a segmented-picker. `MudAutocomplete` with a `SearchFunc` is a find-or-create input. `MudRating` is click-to-save after removing `ReadOnly`. In every case the MudBlazor primitive did exactly what the hand-rolled equivalent would have done, with less code and better defaults. When I started sketching "I'll just build a three-button toggle with CSS…", catching myself and asking "does Mud have this already?" saved half a page of markup. Pattern: for component-library work, default to searching the library's component catalogue before reaching for first-principles UI.
- **View page VM is growing; watch the boundary.** `BookDetailViewModel` went from ~150 lines (PR 1 read-only) to ~280 lines (PR 2 with save methods). Still smaller than `BookEditViewModel` at ~550, and each save method is small and focused. But PR 3 will add modal dialog backing logic for Work/Edition/Copy edits, and PR 4 adds create flows. There's a real risk of the "detail VM" quietly becoming the "everything VM" over the arc. Worth setting a rule now: **if PR 3's modals need >50 lines of logic per entity, they get their own VMs (`WorkEditDialogViewModel`, etc.) rather than methods on `BookDetailViewModel`.** The page VM should own the page's display and the always-visible editable state; modal VMs own the modal's temporary edit state.

## Quotable

The silent-save vs visible-save rule generalises past this page. If the value *is* the feedback — a star rating, a selected status, a toggle position — don't add a "Saved" toast; the UI already confirms by showing the new value. If the user had to *compose* the value by typing content of their own — notes, description, review — then they need to see explicitly that their keystrokes reached the DB, because the textarea itself can't tell them. One rule, applied per-field, removes 90% of the "should this have a Saved toast?" design discussions. The textarea is not the same kind of input as the star picker, even though both live in the same edit flow.
