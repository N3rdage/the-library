## Durable lessons

1. **Commands model intentions, not columns (C10) ŌĆö and the agent defaulted to CRUD.** My first cut decomposed the "mark read" gesture into three field commands; Drew caught it. A command is an *atomic user gesture* ŌĆö neither a field nor a whole screen. Cure was additive: add `MarkBookRead`, keep the field-level commands only where the UI fires them independently (detail-page inline edits). Also: an overloaded *method* serving two intents (`SetStatusAsync` with optional rating/notes) splits into two commands at its two call sites ŌĆö clearer than the overload. See [[feedback_overloaded_field_shared_predicate]] (same altitude family).
2. **The dispatcher arrives at the gate, not the design.** PR0 deliberately said "no dispatcher, inject handlers." When `BookDetailViewModel` hit 10 ctor args, the N-handlers-per-consumer smell was real and Drew flagged it. Fix: a ~40-line hand-rolled `IDispatcher` ŌĆö one injection, O(1) registration. Hand-rolled over MediatR (its recent commercial licence + the "no framework magic we didn't write" ethos). The trigger was *evidence at the gate*, not upfront design ŌĆö don't pre-build the abstraction, but shape the code so it slots in cleanly.
3. **Dynamic dispatch over reflection-Invoke: correct by construction, not by accident.** `MethodInfo.Invoke` wraps a *synchronously*-thrown exception in `TargetInvocationException`; the dispatcher was correct only because every handler happens to `await` before it throws (an unstated invariant). `((dynamic)handler).HandleAsync(...)` propagates the original exception unwrapped AND deletes the `"HandleAsync"` magic string. Constraint surfaced via a *test failure*: handlers must be `public` for DLR binding (they are, by convention). Locked in with a sync-throwing fake-handler guard test.
4. **Adopt-against-existing-tests catches latent bugs for free.** The incremental adoption shape ŌĆö rewire a VM's *internals*, keep its public signatures, re-run the existing behavioural tests ŌĆö caught a real data-loss bug: PR1a's `Book.MarkRead` always overwrote notes, and the pre-existing `ŌĆ”NullNotes_LeavesExistingNotesIntact` test failed the moment the List adopted `MarkBookRead`. The dialog doesn't even surface existing notes, so a blank field would have wiped them. The test was the guard, for free. (Keeping VM signatures stable also meant `Detail.razor` was untouched across PR1b ŌĆö minimal blast radius.)
5. **Aggregate write-boundary discipline keeps the pilot bounded.** Book's write boundary = Book + Editions + Copies; Works/Tags are *separate aggregates the Book references* (their mutations stay in the VM until their own PRs); reads denormalise freely (so `GetBookDetail`'s Work/Series formatters didn't force those aggregates early ŌĆö the read handler was deliberately deferred). The `DeleteBook` split (handler `RemoveRange` vs aggregate clearing joins) was an altitude smell a review caught; fixed by moving `Editions.Clear()` onto `Book.SoftDelete()` (orphan-cascade, same mechanism `RemoveCopy` already used) so the whole soft-delete lives on the aggregate.
6. **(Process) Review subagents need the diff handed to them.** Root-caused during the whole-arc review: a background subagent's *first* Bash call hits "Permission to use Bash has been denied" (no interactive approver), and one finder silently gave up rather than retrying ŌĆö git itself works on later calls. Fix: generate the unified diff in the *main* agent (approved Bash) to a gitignored file and hand finders the path to **Read** (Read never touches the Bash permission gate). This guarantees they review genuine deltas (esp. the `-` lines for removed-behaviour) instead of my summary.

## Surprise

**Three adversarial reviews, zero reachable bugs.** Every "correctness" finding was refuted: clampŌåÆthrow on inputs the rating widget can't produce; "state set even on failure" claims that were factually wrong (the lines sit *after* the `await`); a "lost edition-removal" that was *relocated* onto the aggregate; an "uncaught blank-title throw" that's doubly guarded (Save button disabled + VM guard); a corrupted-`BookId` path that's non-constructible (non-nullable FK). The compiler-enforced `Web ŌåÆ Application ŌåÆ Data` seams delivered exactly the property Drew wanted from the exercise: bounded blast radius, nothing leaking sideways. The real review yield was *altitude/ergonomic* refinements (intent commands, the dispatcher, dynamic invoke, dialog catch+snackbar), not bugs.

## Blog candidates (agentic-coding angle ŌĆö strong)

- **"The agent reached for CRUD; the human caught the altitude."** Task-based vs CRUD commands, and where the human-in-the-loop earns its keep on an agentic build.
- **"The dispatcher arrived at the gate, not the design."** Evidence-driven abstraction vs upfront over-engineering; shaping code so the abstraction slots in when the smell is real.
- **"Correct by construction, not by accident."** Dynamic vs `MethodInfo.Invoke` exception semantics; the unstated-invariant trap in a foundational primitive.
- **"The test you didn't write caught the bug you didn't see."** Adopt-against-existing-tests catching the notes-wipe data-loss bug.
