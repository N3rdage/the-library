## Surprise

- **The AAD broker hang on PR 3 — Microsoft Authenticator intercepting before the redirect.** First sign-in attempt: tap "Sign in" → Authenticator app raised a "Are you trying to sign in to BookTracker Mobile?" toast → tapping it sat there indefinitely. No exception, no log line, no return to the app. Root cause: the redirect URI we'd registered (`msauth://com.thelibrary.mobile/<URL-encoded SHA-1>`) didn't match what Android's intent handler passed back to MSAL after the broker dance, because Android decodes the path component of the intent before matching. The fix was two-part: `.WithBroker(false)` on the `PublicClientApplication` builder to skip the Authenticator broker entirely on this app + switching the manifest path and `WithRedirectUri` to the *raw* base64 signature hash (literal `+` and `=`, not URL-encoded). The symptom was completely silent — no `MsalException`, no Android logcat line indicating mismatch — which made the diagnosis ~45 min of "is the URI wrong, the keystore SHA wrong, the broker config wrong, the intent filter wrong, or the audience wrong?" before isolating.
- **AADSTS65001 on the `az CLI` smoke test that the runbook used to "verify the audience works."** Expected outcome of pre-PR-2 paranoia ("can I prove the audience is configured right before writing any MSAL code?"). Reality: `az account get-access-token --resource api://<clientId>` returned `AADSTS65001: The user or administrator has not consented to use the application` because az CLI's own clientId isn't pre-authorised on the custom API. The runbook's wording matters here — "if AADSTS65001 fires on the smoke test, this is expected and means the audience exists and is consent-gated; either pre-authorise az CLI on Library-Patrons (Step 6 in the runbook) or skip the test and let MSAL.NET ask for consent at first sign-in." Without that line, a fresh runbook reader would think the audience was misconfigured.
- **The Azure Portal redirect-URI editor moved tabs.** Original runbook said "go to Library-Patrons → Expose an API → Redirect URIs." The actual current path is buried under "APIs my organization uses," and the entry is matched by clientId GUID, not display name. Runbook updated mid-arc.
- **PR 4's NRE was in the success-message line, not the populate path.** Symptom: tap "Load catalog snapshot" → "Object reference not set to an instance of an object" → no other detail. Initial fix attempt was defensive null-coalesce in `PopulateAsync` (correct fix for general robustness, but didn't move the symptom). Second attempt with a stack-frame-in-the-label diagnostic showed the throw was in `OnLoadCatalogClicked`'s result-message string interpolation: `snapshot.Series.Count` where `Series` was null because the API server hadn't been redeployed yet to include the new `series` field added in PR 1. Final fix: `?? 0` everywhere in the success message + a server-staleness hint line. **Diagnosis lesson:** when the user-visible NRE has no stack trace, paying ~2 minutes to capture the throw frame is worth it. The defensive null-coalesce was the right code change; it just wasn't the line that was throwing.
- **sqlite-net-pcl treats the connection path as a literal filename, not a URI.** PR 4's test suite tried `:memory:?cache=shared&mode=memory` for per-test isolation. Result: `Could not open database file: :memory:?cache=shared&mode=memory`. sqlite-net-pcl's path-based connection pooling means the path string is what indexes the pool, and it's passed to SQLite open as-is. Two options: (a) plain `:memory:` (but then per-test isolation fails because the same in-memory DB is shared); (b) per-test GUID-named temp files. Went with (b) — `Path.Combine(Path.GetTempPath(), $"booktrackermobile-test-{Guid.NewGuid():N}.db")`. Tests now isolated, ~12ms each.
- **The polish PR (#6) was a single-file XAML edit and built clean first try.** After the four-iteration NRE diagnosis of PR 4, expectations were calibrated to "the next thing will surprise me too." It didn't. The overlay went in as a static Grid-in-Grid layering, star sizing worked, build produced zero new warnings, on-device verification was instant. Worth noting because not every PR is a struggle — when the design is borrowed and the tooling is straightforward, sometimes it just works.

## Lesson

- **The arc was fast because every design choice was already paid for somewhere else.** This is the load-bearing observation. ~20 hours wall-clock (8h active) covered: a new project type (MAUI Android), a new auth flow (AAD interactive on mobile vs Easy Auth cookies in browser), a new persistence layer (sqlite-net-pcl), a new camera library (ZXing.Net.MAUI). None of that work *invented* anything. The catalog DTO existed (PR 1 extracted it from Web's internals into `BookTracker.Shared`). The debounce-and-lookup pattern existed (`/bookshop`'s `barcode-scanner.js`). The alias-rollup query existed (`AuthorService.GetWithAliasesAsync`). The Easy Auth setup existed (just needed audience-mode toggle, not redesign). The JSON enum-as-string convention existed (`ConfigureHttpJsonOptions` was added during the bookshop arc). The mobile arc was *applying* design decisions, not *making* them. **Generalises to:** if you're considering a second app (mobile, CLI, second tenant, white-label), the design tax was paid by app one. The work isn't "build app two from scratch" — it's "ship app two using app one's decided answers." That changes the time estimate by an order of magnitude.
- **Bookshop and mobile are two solutions for two problems.** When the `/bookshop` arc was conceived, the design was "one tool that handles both at-home browsing AND in-bookshop scanning." It shipped useful — Drew was using it in real bookshops the same week, via the existing Blazor app's PWA. But carrying it around exposed that the in-bookshop case wanted a different container: offline-first, AAD-tokens-on-device, native-camera, no SignalR circuit assumption. The pivot at 2026-05-10 19:36 (design doc merge for the mobile arc) didn't *replace* bookshop; it recognised that the workload was actually two workloads. Bookshop stays as the no-install, anyone-with-the-Web-app gets it tool — useful in its own right. Mobile is the right shape for the case bookshop was stretching to cover. **Generalises to:** when a tool is being asked to do two adjacent jobs, the right question isn't "how do I make this tool better at both" — it's "are these actually one job or two?" Shipping the stretched-version of the tool is often what makes the answer clear, because real usage tells you which parts strained.
- **`BookTracker.Mobile.Cache` as net10.0 (not net10.0-android) is the pattern.** The cache project doesn't need any MAUI types — it's plain SQLite + DTOs + LINQ. Splitting it from `BookTracker.Mobile` (the MAUI project) means unit tests run via `dotnet test` without an Android device or emulator. Five minutes of csproj setup; saves the entire "spin up emulator to run a unit test" tax that's traditionally killed mobile testing. **Generalises to:** any logic that doesn't strictly need the mobile runtime belongs in a sibling project that doesn't have the mobile runtime as a dependency. Keep the runtime project thin — UI + DI wiring + lifecycle handlers; nothing else.
- **Real device every PR, no emulator.** Drew tested each PR on his actual phone before merging. Worth the friction (~3 min to deploy + open the app vs ~30s for emulator) because the AAD broker bug and the camera-permission flow are exactly the kind of things that work on emulator and fail on device. The first time emulator-only would have been "saved 15 minutes" was the bug that took 45 minutes to diagnose on device anyway. **Generalises to:** for any platform where there's a real-vs-emulated gap (mobile especially, but also: AAD federated identity, OAuth refresh flows, network conditions, IDLE timeouts) — the real environment is the test environment, full stop. Emulators are a development-loop accelerator; they're not a validation surface.
- **The "small follow-up PR for polish" discipline kept each PR shippable.** The mobile arc easily could have been 1 PR ("the mobile app"). Instead it was 5 PRs of strict layering — shared DTOs, then docs, then skeleton, then cache, then scan — followed by a 6th PR that was just visual polish (letterbox UX). Each PR had a clear "this works end-to-end at this layer" success criterion. PR 4 shipped with the result panel as plain text — no green-on-found, no neutral-on-missing. That was added in PR 5 as part of the scan flow. The letterbox came in PR 6. **Generalises to:** when you're tempted to ship "almost-done" with the polish lumped in, lump *less*. Ship the working core, then ship the polish as its own PR with its own commit message. Future-you can grep "letterbox scan" and find one diff, not a 600-line PR with letterbox buried at line 412.
- **`Mobile.Cache` as a separate project + targeting net10.0 also meant the tests could run *fast*.** The Mobile.Cache test suite is 9 tests, ~120ms total. Drew can run them in a tight feedback loop while iterating. No emulator boot, no MAUI workload load, no Android SDK in the test runner. Companion point to the previous lesson, but worth its own line — fast tests aren't a nicety, they're the difference between "I'll run the suite" and "I'll skip the suite this time, it's only a small change."
- **Build warnings, not just errors, deserve a clean state.** PR 5's first build had 15 warnings — most were `Frame` obsolescence (replaced in .NET 9+), plus a nullable-mismatch on the event handler, plus 4 unfixable upstream XA0141 page-size warnings from camera-core and SQLitePCLRaw. The discipline that paid off: fix the ones that *are* our code (Frame → Border, `object` → `object?`) so the remaining noise is *only* upstream, and a future regression introducing a new warning is visible against a clean baseline. Without that discipline, "0 errors, 15 warnings, build succeeded" silently accumulates regressions in adjacent code.

The headline lesson, blog-shaped. Speed wasn't going faster; it was not having to invent anything. Catalog DTO existed. Debounce timing decided. AAD already secured the API. > "The AAD broker hang was silent. No exception, no logcat line — just the Authenticator toast that wouldn't go away."

Diagnosis story worth remembering. The fix was URL-decode the redirect URI to its literal `+`/`=` form, drop `.WithBroker(false)` if the Authenticator broker is what's intercepting. The URI is the part Android touches before the auth library gets it back.
