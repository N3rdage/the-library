## Surprise

- **The silent `string.Join(record)` bug had been in `main` for a day.** When `AllAuthors` switched from `IReadOnlyList<string>` to `IReadOnlyList<AuthorContribution>` on 2026-05-23, the mobile `ScanPage` site `string.Join(", ", book.AllAuthors)` compiled clean and ran clean — the generic `string.Join<T>(string, IEnumerable<T>)` overload was selected, `T.ToString()` was called on each `AuthorContribution`, and the record's auto-generated `ToString` returned `"AuthorContribution { Name = X, Role = Author }"`. No exception, no crash, no test failure — just garbage on the screen. Drew didn't notice for ~24h because he wasn't scanning multi-author books in that window. **The fix landed accidentally** as part of Bucket 2's `ContributorFormatter.Format` rewrite — the new code path replaced the broken one because both happened to render the same field. If Phase E had been deferred past this point, the bug would have stayed latent until the next Bookshelf scan of a multi-author book. **Generalises:** wire-shape renames cross language boundaries silently. C# generic overload resolution + record `ToString` is a particularly nasty combo because the call site compiles, runs, and "succeeds" — there's no syntactic signal anything is wrong.
- **Bucket 1's plan dissolved the moment Drew named the real pain.** The pre-arc memory described Bucket 1 as "do a coverage-sampling pass first, then commit to the OL parser." I opened the conversation asking which sampling shape Drew wanted; he said "skip it" and named the Author-mandatory problem instead. The work that shipped was 10× higher value than the planned work would have been — the parser would have addressed a *capture-time gap that doesn't exist yet* (reference books haven't been mass-captured), whereas the editor-only relax addressed a capture-time pain Drew was *already feeling on every dictionary*. The lesson is not "don't plan" — it's "the plan is a starting point, and the user knows the real pain."
- **TODO #48 had silently half-shipped.** The drift probe during the sync flagged it: `/api/wishlist-snapshot` + `WishlistSnapshotService` + the Shared/Wishlist DTO are all in `main`. The TODO row still said "backend shape decided 2026-05-13" — past tense for the decision, present tense for the work that had also already happened. Without the post-shipping probe, Bucket 4 or 5 of the next planning session would have been "implement the wishlist backend" and the first commit would have surfaced "huh, this already exists." Same shape as the 2026-05-13 sync that caught four shipped-but-unmoved rows.
- **Adding a `BookStatus` value cost zero schema work.** The enum is stored as `int`; existing rows can't hold the new value (3); the snapshot wire field is already `string` ("Reference" just flows through). It was easy to *expect* a migration there — Edition.EditionNumber needed one, why wouldn't BookStatus.Reference? — but enum-extension on an int-stored enum is free unless you're adding constraints or check constraints. Worth remembering: not every "schema-shaped" change needs a migration.
- **One PR's tests stay green while another PR's tests need updating.** Bucket 1's relax broke `SaveAsync_NonCollection_NoAuthors_ThrowsWithUserFacingMessage` (renamed to `..._NoContributors_...` and assertion shifted from "author" to "contributor"). Bucket 2's `string.Join` bug had *no* test catching it — the multi-author display path had no bUnit / cache coverage because it was pure UI rendering. Bucket 3's enum addition broke nothing. The test-debt heat map across the three PRs lined up roughly with the surprise count: highest-tested work had fewest surprises, lowest-tested work had the silent-failure-for-a-day bug.

## Lesson

- **Bucket plans are throwaway once the user names the real pain.** The pre-reference-capture-arc memory had a clean three-bucket ordering. Within five minutes of the first "what's next?" conversation, Bucket 1 was redirected, and the redirected work was higher-value than the original plan. The lesson isn't "skip planning" — it's that the plan's job is to put the user *in front of* the pain so they can name it. Once the pain has a name, the plan exists to be discarded. Worth banking as a feedback memory if it recurs (it has at least once before in the Add/View/Edit polish arc — see PR 5/6 there where Drew steered to "Ask, but allow free-text" away from the proposed two-button design).
- **Compile-clean is not runtime-clean across wire renames.** The `string.Join(", ", AllAuthors)` site survived a type change from `IReadOnlyList<string>` to `IReadOnlyList<AuthorContribution>` without a single compile warning, *because the generic overload exists and record `ToString` exists*. The defense — a runtime smoke test against a known multi-author book on Bookshelf — would have caught this in the first deploy. Not worth writing one *just* for this case, but worth banking the generalisation: **type renames that cross runtime boundaries (server → wire → client) need a runtime sanity check, not just a compile.** Already captured indirectly in [[feedback_hot_reload_commit_gap]] (hot-reload ≠ committed) and [[runbook_container_warmup_calibration]] (cold path differs from warm) — this is a third member of the same family.
- **Sync-probe ROI is high.** The 2026-05-13 sync caught four shipped-but-unmoved rows; today's sync caught two more (TODO #48 backend, ProgramSetup.cs stale comment). The pattern: anything that ships *outside the primary capture loop* — endpoints with no consumer yet, helper code, comments — drifts because there's no user-facing flow to remind anyone. A periodic sync is the cheap recovery mechanism. Cost: ~10 min of agent-driven probing per cycle. Catches days of "go ship X" wasted work.
- **Shipping the scaffolding for an off-platform activity feels small but compounds.** None of today's three PRs has a single new user-facing feature in the catalogue-level sense — the user can't point to anything they can do today that they couldn't do yesterday, *unless* what they're about to do is mass-capture a hundred reference books. Then every PR pays. The retrofit cost (re-keying every reference Edition's revision number, hand-editing every dictionary's "Unknown" author, re-categorising every Reference status) is invisible up front and stings only at scale. Worth holding the framing: **scaffolding for the user's actual work is feature work, even when it doesn't look like it on the diff.**
- **Three small PRs > one mega-PR even when the buckets are paired.** Buckets 2 + 3 could have been one PR ("everything reference-capture needs on the data side"). They weren't — and the split made each one easier to plan, easier to review, easier to revert if something landed wrong. The seams (mobile cache vs data model) were natural. Same lesson as the Add/View/Edit arc: **when a feature has a natural seam, the seam is cheaper than the integration test.** This is the third time this lesson has surfaced in a retro; promote it.
