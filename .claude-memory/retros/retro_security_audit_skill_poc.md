## Surprise

- **The `.claude/` gitignore footgun is real and the skill's default path was wrong.** The skill loads project rules from `.claude/audit-rules/security.md`. BookTracker tracks `.claude/` (its `.gitignore` only ignores `.claude/settings.local.json`), so project rules + suppression rationales travel with the code. what-now's `.gitignore` (from `create-next-app`) ignores all of `.claude/` — the more common default. That means in any "normal" repo, project-level suppressions would silently be per-machine: a teammate (or future-me on a new clone) would run the audit and get different findings, with no record of *why* a rule was suppressed in the first project. **The diagnostic surface this exposed: when designing chassis-shaped tooling that reads project files, audit your assumed paths against the *common* repo defaults, not the project that happens to be in front of you.** BookTracker was the outlier; what-now was the canary.
- **The chassis carried first try across two very different stacks.** The fear with "stack-specific templates layered on cross-stack defaults layered on project rules" was that the structure would feel right on the project that informed it (BookTracker — designed against this exact shape) and then need re-shaping on the second stack. It didn't. The Next.js report came out *legibly* the same as the BookTracker report despite different rule sets, finding shapes, and a project that had no project-rules file at all (template + defaults only). Severity tally, finding citations, "Verified clean → Suppressed → Detail" structure all carried. That's what "chassis" was supposed to mean and the fear was that it wouldn't earn the word.
- **"No applicable surface yet" is a third pass-with-evidence shape I hadn't planned for.** The two I designed for were "verified clean — here's the grep / Bicep ref / file inspection" and "documented concession — here's the rationale and the deferred-items link." what-now's run added a third: NEXT-001 (RLS on user-data tables) ran and recorded "no surface — there are no user-data tables in this project yet, this rule will activate when Supabase migrations land." That's a rule paying for itself before the surface exists: the next audit, post-Supabase, will surface drift if a table is created without RLS. The chassis is richer for this shape; future audit skills (`a11y-audit`, `scale-audit`) should expect it.
- **The previous Claude session got a rule ID wrong when offering setup advice and Drew caught it.** I'd written "if there's no Storage usage yet, NEXT-006 storage-bucket policies isn't relevant" as an example of suppression. The actual NEXT-006 is the magic-link redirect URL allowlist; there's no storage rule in the template at all. Confidence-without-grounding bug — the example sounded plausible enough to ship without checking the file. The fix isn't "more rules"; it's that suggestions about *specific* rule IDs need a verification step before the answer leaves my mouth, even when the surrounding advice is correct.
- **Skill-found-but-doc-missed and doc-found-but-skill-missed are both useful drift signals.** I ran the comparison expecting one direction to dominate. Both directions surfaced gaps: skill caught `SEC-011` open redirects, `SEC-012` CORS, `SEC-013` rate limiting, `AZRE-008` OIDC FIC scope, `BOOK-002` empty-staging migration risk — the living doc had no sections for these (true clean items, useful to fold into the doc). Doc had a Blazor `IJSRuntime.Invoke*` JS interop check (`SECURITY-AUDIT.md §9`) that the skill's `SEC-010` doesn't grep for — that's the gap to close as `AZRE-009`. The doc + skill complement each other; neither alone catches the full set.

## Lesson

- **Two-stack validation is the bare minimum for "chassis" claims; one stack proves the rules, two stacks prove the structure.** With only BookTracker, the skill was a security audit checklist. The what-now run answered the actually-load-bearing question: does the same shape produce useful output on a different surface? You can't answer that with a single project, no matter how well it's designed. Generalises to the audit-skill family roadmap (`scale-audit`, `a11y-audit`, `codehealth-audit`): each one needs defaults + at least two stack templates + runs on real projects from at least two stacks before the "chassis" word earns its place. One-stack work is a bespoke skill; two-stack work is a chassis pilot.
- **Project-level policy files belong outside `.claude/`.** `.claude/` is conventionally where Claude Code's per-machine config lives — `settings.local.json`, MCP server keys, IDE-specific bits. Project audit policy isn't *per-machine*; it's *per-project*, and it should travel with the code on first clone. The same logic that puts `.editorconfig`, `.gitignore`, `CONTRIBUTING.md`, and `CODEOWNERS` at repo root applies to `audit-rules/`. The skill currently puts project rules at `.claude/audit-rules/security.md` because that's where BookTracker landed; the durable home is `audit-rules/security.md` at repo root. Two projects to migrate manually (BookTracker + what-now); skill files to update so the canonical path moves and the templates' setup notes recommend the new location. Lesson generalises: when picking a project-level config path, check what `.gitignore` does to it in the common case (`create-next-app` output, `npx create-react-app`, `dotnet new web`, etc.) — if the answer is "ignored by default," your config will silently be per-machine.
- **Pass-with-evidence has at least three shapes; the chassis report-template should accommodate all of them explicitly.** "Verified clean" (rule ran, surface exists, surface is fine), "documented concession" (rule ran, surface exists, finding is the deliberate trade-off), "no applicable surface yet" (rule ran, surface doesn't exist yet, recording this primes the next audit). The third shape is what makes audit reports useful as forward-looking trip-wires rather than just point-in-time scans. The skill's `report-template.md` should be updated to make all three first-class in the "Areas verified clean" section's structure.
- **Suggestions about specific rule IDs / line numbers / file paths need verification before they leave the assistant's mouth.** The NEXT-006 mistake was a confidence-without-grounding bug; I had the *kind of advice* right (suppress what isn't relevant) but invented a specific example without checking the template. The fix isn't a process step — it's recognising that "specific identifier" is a different epistemic class from "general principle" and the bar for the former is "I just looked." Applies to any suggestion that names a function, a config key, a file path, a rule ID: if I can't point at the line, I haven't earned the right to cite it. Better to gesture at the shape ("suppress whichever NEXT-* rules don't apply — pick from the template's section headers") than invent a wrong specifics.
- **Living docs and per-run reports are complements, not duplicates; cross-checking them surfaces gaps both directions.** The chassis pilot was the first chance to run that cross-check on the same project. Outcome: skill caught five things the living doc didn't have sections for; doc had one Blazor-specific check the skill didn't grep. Going forward, the validation-cycle pattern is: run the audit, diff against the living doc, record drift in both directions, decide whether each drift is "doc needs section" or "skill needs rule." The living doc keeps human-facing context (rationale for concessions, deferred-items roadmap); the per-run report is the trend record (file:line citations, frontmatter for machine processing). Future audit skills should expect this same complement.

## Quotable

The chassis-validates moment was less about whether the *rules* worked (they did, 47 of 47 across two runs) and more about whether the *shape* did. The shape is: cross-stack defaults that any project can run, stack-specific templates that auto-load when the project type matches, project-level rules + suppressions that override either layer with rationale, and a report structure that's legibly the same regardless of which combination ran. The proof is that the BookTracker report and the what-now report read like siblings — same headings, same evidence shape, same severity treatment — even though one has 24 rules with three layers and three findings, and the other has 23 rules with two layers and two findings. **Chassis means the structure carries, not just the content.** Once that holds, the same shape extends to `scale-audit`, `a11y-audit`, `codehealth-audit` for free; without it, every audit skill would be a one-off and the family wouldn't be one.

The `.claude/` gitignore lesson is the more humbling one: I designed the skill assuming the BookTracker shape (`.claude/` tracked) and didn't notice until what-now (`.claude/` ignored, the more common default) ran the audit and exposed the implicit assumption. **The project that informs the design is rarely the project that exposes the design's flaws.** That argues against single-project pilots for chassis-shaped work in general — not because the first project is wrong, but because it's *representative of itself* and the second project's job is to be representative of *not that*. Two stacks isn't a nicety; it's how you find the assumptions you didn't know you'd encoded.
