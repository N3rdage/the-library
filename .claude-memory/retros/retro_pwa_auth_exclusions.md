## Shipped

PR #92 — 2 files, two commits. Added `globalValidation.excludedPaths` to the prod + staging `authsettingsV2` in `infra/modules/app-config.bicep` so PWA install validation can reach the manifest, service worker, and icon files without an AAD sign-in redirect. Also added a Security Audit item to `TODO.md` for a broader posture sweep.

## Surprise

- **Easy Auth v2 `excludedPaths` does EXACT match, not prefix match.** I assumed prefix matching (because that's how most URL-matching config works and nothing in the Azure docs is explicit). First pass listed `/icons` — worked for requests to `/icons` directly but not `/icons/icon-192.png`. Chrome's PWA install validation fetches each referenced icon uncredentialed, so three of the four icons still 302'd, and install stayed broken. Fix was to enumerate all four icon files explicitly. Dropped a big comment warning future-me that new icons need a matching exclusion entry.
- **`az webapp auth show` returns a V1-shaped view even when V2 is running.** The output has `configVersion: v2` but the properties are flat (`clientId`, `allowedAudiences`, etc.) — none of the V2 structure (`globalValidation`, `identityProviders`, `platform`, `login`). Tried to grep for `excludedPaths` and it wasn't there, which made it look like the Bicep deploy had silently dropped the field. Wasted a diagnostic round chasing that before switching to the REST API: `az rest --method get --uri "https://management.azure.com/.../config/authsettingsV2?api-version=2023-12-01"` returned the real V2 JSON, which clearly showed `excludedPaths` was in fact deployed. The CLI is legacy and hasn't caught up with V2.
- **PWA install validation fetches manifest and icons without cookies.** The `<link rel="manifest">` fetch (and Chrome's subsequent icon-existence probe) don't default to credentialed mode. Behind an authenticated site, this is a fundamental mismatch that always needs a bypass for those specific paths. Not a bug, not a misconfiguration — just the spec. Any Azure-auth-gated app doing PWA will hit this.
- **Running `deploy.ps1` from the feature branch was the right workflow.** Drew ran it pre-merge to test the fix, which surprised me initially because "deploy from feature branch" feels taboo. But `deploy.ps1` reads the Bicep from the local checkout, not from main, so deploying from the feature branch is equivalent to a preview environment — and reverts automatically when main's deploy.ps1 runs later. Actually a nice workflow for infra iteration where you don't want to merge a change only to then find it's broken.

## Lesson

- **Match semantics for ACL-ish config are under-documented and you should always enumerate explicitly first.** Prefix matching *feels* like the right default and *is* the right default for most systems (Nginx location, AWS IAM resources, etc.). Easy Auth doesn't do it. When I write this kind of exclusion/allow list next time, first draft should enumerate exact paths even if it's ugly, until I have documented confirmation of the match semantics. Ugly-but-deterministic beats elegant-but-broken every time, especially for security boundaries.
- **Azure CLI version drift is a diagnostic hazard.** `az webapp auth show` is a legacy command that pretends to work against V2 configs but silently omits V2-specific fields. When the CLI output looks suspiciously sparse, hit the REST API directly. Pattern to remember for future Azure diagnostic cycles: if `az <service> <action>` returns something that doesn't match the resource shape you expect, ARM `az rest --method get --uri ...` is the ground truth.
- **PWA + auth gate = known conflict. Bake the exclusions in from the start.** I shipped PR #91 with full Easy Auth in effect and didn't think through the uncredentialed-fetch path. Cost: one round of "why doesn't this work" from Drew + a follow-up PR + a re-deploy. If I'd set this up from scratch today I'd include the `excludedPaths` in the original PWA PR as a matter of course. Filing this as a forward-looking note: any feature that involves a browser-level handshake (PWA install, OAuth callbacks, webhooks, external-service callbacks) touching an auth-gated app needs an explicit bypass planned upfront.
- **"Test from the feature branch before merge" is a legitimate pattern for infra PRs.** I initially flagged it as skipping review, but the actual flow — deploy from feature branch → test → merge if good, revert by re-running deploy from main if bad — is safer than merging-first-then-testing-in-prod. Worth encoding as a durable pattern for infra changes. Different shape from app-code PRs where CI provides most of the pre-merge validation.

## Quotable

Three rounds of diagnostic work for what ended up being a six-line config change. First round: I proposed the wrong match semantics (prefix instead of exact). Second round: the Azure CLI lied about what was actually deployed. Third round: the fix itself, delivered in under a minute. The real cost of infra bugs isn't the fix — it's the credibility of your diagnostic tools. When the CLI returns sparse output for a fully-configured resource, you don't trust anything the CLI says for the rest of the session. The REST API is the thing that actually knows; every other layer is an abstraction with its own failure mode.
