## Durable lessons (beyond the pilot's six)

1. **Verbatim-relocation discipline, and its deliberate inversion at close-out.** The arc's spine rule: a behaviour-preserving move *inherits* the legacy code's smells (wrong altitude, in-app rollups, cartesian Includes) — **log them (→ TD-17), don't fix mid-relocation**, because a "fix" changes behaviour and breaks the no-op guarantee the whole arc rests on. The close-out is where that discipline inverts: the consolidation (`AuthorRollups`/`GenreReads`/`BookCovers`) is the *only* hand-authored code in the arc, so Drew's sequencing call — pay down the logged TD *before* the whole-arc review, not after — put exactly that novel code under the most adversarial pass. Schedule your hardest review for the code you wrote fresh, not the code you moved.

2. **Every reachable bug was pre-existing latent — the refactor's value was exposure.** Across the arc, every confirmed bug an adversarial review caught was a *pre-existing* defect the relocation surfaced by forcing you to name what the old code did: PR1 notes-wipe data-loss, PR3 stranded `SeriesOrderDisplay` after series-removal, PR4 stale-in-memory-row duplicate book on double-click, PR6b-3 tag-casing flip on reload. Zero were *introduced* by the new structure (one — 6b-3 — was *introduced by the move re-deriving a value the old code read off the entity*, the instructive exception). "Behaviour-preserving" is the lens that makes latent bugs visible.

3. **DRY would have been a bug — "duplication" ≠ "the same thing written twice."** The original plan said "extract a shared author-rollup query." Inspection showed Home's top-authors (contribution tally), `/authors` (distinct works/books/series), and the snapshot `BookCount` (distinct books, a cross-app contract) were **three deliberately-different metrics**; collapsing them would silently change Home's headline. Kept as three, logged the drift risk (TD-17), and at close-out converged them *on purpose* with Drew's explicit "display-count semantics may change" licence — the convergence was a product decision, not a refactor reflex. See [[feedback_light_reads_custom_projections]].

4. **The ORM wall, and the correlated-subquery escape.** EF Core 10 refuses to translate both `Distinct().GroupBy()` and `GroupBy().Select(g => g.Distinct().Count())` (a literal `GROUP BY COUNT(DISTINCT)`). The shape that *does* translate and keeps the read light is a **per-row correlated `COUNT(DISTINCT)` subquery** (`a.WorkAuthors.Where(Role).SelectMany(wa=>wa.Work.Books).Select(b=>b.Id).Distinct().Count()`) — one row per group, the aggregate pushed into a subselect. The kicker: that's the exact shape the legacy `directCounts` already used. When the ORM walls your group-by, push the aggregate *down* into a correlated subquery — and check what the code you're deleting already did. Captured in [[feedback_light_reads_custom_projections]].

5. **"EF won't translate it" talks you into the compromise — a second pass catches it.** Close-1's first cut retreated to materialising ~3.6k (author,book) pairs + in-memory group-count *because* the SQL group-by wouldn't translate. Invisible at 2k books on the cold `/authors` page; a real regression on Home + the snapshot (read every load). The whole-arc review flagged exactly that, and only then did the correlated-subquery shape surface. The wall is a real constraint, but it's a reason to escalate the query shape, not to silently accept an O(rows) transfer on a hot path.

6. **Risk tracks the write boundary, not the diff size.** The arc's biggest-LOC PRs (BookList, Wishlist read relocations) were the *lowest-risk* per line — every write had already been lifted in PR1–5, so these were pure read moves that can't lose data, only mis-shape a projection the tests catch. The small write PRs carried all the danger. Size your caution by what a slice can *break*, not how much it touches.

7. **Convention-scan DI makes the integration test layer auto-track relocations.** VMs build a real dispatcher via `AddApplicationLayer()` (convention scan). The instant a relocated `Get*` handler *exists*, it's registered and the existing VM integration tests exercise it end-to-end — zero new wiring. The relocated reads were "covered" the moment the VM dispatched to them; this is what let heavy incrementally-migrated VMs (BookDetail, BookList) keep their integration suites as handler-wiring tests instead of re-cutting to mocks.

8. **The read handler pulls its formatters with it.** Four pure display formatters (`WorkAuthorshipFormatter`, `SeriesOrderParser`, `PartialDateParser`, `BookFormatExtensions`) migrated Web→`Application/Formatting/` over the arc — each dragged across by a read handler needing to *project* a formatted field (C5: the handler owns the DTO including its formatted strings). You can't half-move a read; the projection and its formatting must live on the same side of the boundary.
