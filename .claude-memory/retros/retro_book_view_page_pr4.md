## Shipped

PR #99 — four new edit surfaces reachable from the View page:
- **Add edition** — ISBN field with Lookup button (`IBookLookupService` auto-fill-empties-only), format, publisher typeahead (same find-or-create shape as Author), date printed (`PartialDateParser` free-text), cover URL, first copy's condition.
- **Edit edition** — same fields, no Lookup, no first-copy.
- **Add copy / Edit copy** — condition, date acquired (`MudDatePicker`), notes.

Delete copy uses inline confirm/cancel rather than modal-on-modal. Deleting the last copy on an edition cascades the edition removal (matches existing `/edit` behaviour — an edition with zero copies doesn't mean anything). Two dialog VMs (`EditionFormDialogViewModel`, `CopyFormDialogViewModel`), each covering Add + Edit via an `IsNew` flag. `BookDetailViewModel` gained one method (`DeleteCopyAsync`); otherwise unchanged.

The page is now feature-complete for normal single-Work books. What's still on `/edit`: genres (hierarchical MudBlazor rebuild is a dedicated future PR) and compendium building (Add-other-works / Attach-existing-Work).

## Surprise

- **EF InMemory is case-sensitive; SQL Server default collation is case-insensitive. A test caught the divergence.** `SearchPublishersAsync` was written as `db.Publishers.Where(p => p.Name.Contains(q))`. The test seeded `["Corgi", "Gollancz", "Orbit"]` and searched for `"or"` — expected three matches, got one (just "Corgi", whose lowercase substring contains "or"). In production against SQL Server with default case-insensitive collation, the same query would have matched all three. This is a genuine "works in prod, broken in tests" / "works in tests, surprising in prod" footgun. Fix: explicit `.ToLower()` on both sides of `Contains` inside the query. Applied to both Publisher search and the pre-existing Author search (which had the same latent bug from PR 3 — nothing had exercised it cross-case). **Lesson with teeth: never rely on collation when testing against InMemory. If your production is case-insensitive and your test is case-sensitive, the test's negative assertion ("doesn't find Orbit") is the one that silently breaks.**
- **Dual-mode dialog VMs with `IsNew` flag scaled well when ~80% of the fields overlap.** `EditionFormDialogViewModel` and `CopyFormDialogViewModel` both cover Add and Edit through a single `bool IsNew`. The differences are narrow (Add shows Lookup button + first-copy field; Edit skips those), so a flag + a few conditional Razor blocks beats four separate VMs. I'd considered splitting when planning — this validates staying combined. Rule: if the delta between "Add" and "Edit" variants is fewer than ~20 lines in either VM or Razor, one VM + flag; more than that, split.
- **Variable shadowing errors in tests (`CS0136`) are a recurring minor annoyance.** I reused `edition` and `copy` as local variable names in two scopes of the same test method (one in the seeding `using` block, one in the assertion phase). C# won't let you do that even when the scopes are separate. Classic "I wrote the 'obvious' name in both places and the compiler complained". Fix was trivial (rename to `seedEdition` / `seedCopy`) but I keep making this same mistake. Small tax, not worth over-engineering a solution for.
- **The ISBN Lookup auto-fill-empties pattern ported from `BookEditViewModel` unchanged and still feels right.** Users can pre-type values they know (format, date, cover URL), hit Lookup to fill gaps, and their typed values stay intact. If the user wants to wholesale overwrite from the ISBN match, they can clear the field first, then look up. The "clobber nothing" default is the right one — overwriting is surprising; preserving is expected. This pattern generalises to any "enrich from external source" flow: always fill gaps, never overwrite.
- **The inline delete-confirm is the right shape, proven on first contact.** Clicking Delete turns the button row into `[Delete this copy?] [Delete] [Cancel]`. No modal-on-modal, no separate confirmation dialog, no "are you sure?" prompt overlay. It's local to the row and matches the existing `/edit` page's pattern. Mobile-friendly (everything reachable with thumb-tapping; no dialog pushing other UI out of the viewport). Generalises: destructive actions inside a list of items should confirm inline, not in a new modal.

## Lesson

- **Case-insensitivity must be explicit in LINQ when InMemory is in the test loop.** Don't rely on the production provider's collation to absorb case variations. Write `x.Name.ToLower().Contains(q.ToLower())` (or use a configured `EF.Functions.Like` / case-folded computed column if it comes to that). Yes this defeats any index SQL Server might have on `Name`; at this project's scale (low thousands of rows for Authors + Publishers, maxing at tens of thousands of tags), the table scan is fine. If we ever need a case-insensitive index, add a persisted computed `NameLower` column. For now: explicit client-side lowering is the portable, predictable shape.
- **Dual-mode VMs (Add+Edit via flag) have a specific threshold for when to split.** ~20-line delta between the two variants is the rough trigger. Below that, the flag is cleaner. Above it, you start reading conditional branches just to understand any single method, and the VM is secretly two VMs in a trench coat. The Edition dialog sits comfortably below the threshold (delta is ~15 lines across VM + Razor — Lookup button visibility, first-copy field, the SaveAsync branch). If a future dialog has meaningfully different flows for Add vs Edit, split at design time, not after.
- **Inline confirm-cancel is the right pattern for destructive actions inside lists.** Modal-on-modal (`Are you sure?` inside a dialog that's already open) is worse: the nested dialog breaks focus flow, can't be tab-navigated cleanly, and on mobile it covers the item you're about to act on. The inline pattern keeps context visible. Stockpile this decision for anywhere users delete items from a list.
- **Port patterns before re-designing them.** The ISBN Lookup auto-fill-empties logic ported from `BookEditViewModel.LookupNewEditionAsync` almost line-for-line. I briefly considered "is this the right shape?" and the answer was yes — it was working, it generalises, there's no reason to redesign. Applies to the publisher find-or-create pattern (ported from AuthorResolver's usage), the inline delete confirm (ported from the existing Edit page), and the PartialDateParser free-text date field (ported from BookEdit + WorkEdit). If the existing pattern is working, reuse it verbatim before mining for improvements. The bar for "let me redesign this" should be higher than "it's adjacent to something I'm touching."

## Quotable

The EF InMemory case-sensitivity divergence is one of those invisible test-vs-prod gaps you can run against for months without noticing. The fix is one line of code (`ToLower()`); the lesson is bigger: any cross-provider ORM test strategy has implicit behaviour differences, and relying on your prod provider's quirks is the shape of all the worst bugs. SQL Server's case-insensitive default collation is silently doing UX-important work. Move that work into your query, not out of your test harness. The general rule: **if your test provider behaves differently from your prod provider on a behaviour you rely on, make the behaviour explicit in code instead of wishing it away.** The test that caught this wasn't clever — it just seeded three publishers with mixed cases and asked for a substring. The cleverness was writing tests that could surface the gap at all.
