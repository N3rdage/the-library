## Shipped

PR #129 — TODO #1 (Separate staging database from production) delivered as one PR. Closes the deployment-safety gap where a destructive migration on staging took prod down before any swap, and slot swap-back was never a real rollback (old binaries against new schema).

- **Bicep:** added a second `Microsoft.Sql/servers/databases` resource for staging on the same SQL server (`booktracker-staging`, Basic tier), piped through `resources.bicep` and `main.bicep` outputs.
- **`app-config.bicep`:** staging slot's CS now points at the staging DB; added `DefaultConnection` to `slotConfigNames.connectionStringNames` so the CS is slot-sticky. Without that pin, a swap would land the formerly-staging code on the prod URL still pointing at the staging DB — an immediate prod outage.
- **`deploy.ps1`:** AAD grant runs per-DB / per-identity (prod identity → prod DB only, staging identity → staging DB only). Drops orphan staging-identity grants from the prod DB on re-runs so a slot-sticky-CS regression would surface immediately rather than silently working.
- **Docs:** `infra/README.md` got a "Staging is a real environment now" section calling out the empty-staging caveat (catches schema syntax errors, misses every data-shape failure) and an explicit order-of-operations note for the first run. `CLAUDE.md` and `ARCHITECTURE.md` got the connection-string convention update.
- Cost: +~$5/month for the second Basic-tier DB.

Two follow-up TODOs surfaced and landed in the same housekeeping PR (#?):

- TODO #12 — Health monitoring + perf budgets + capacity alerts (synthetic heartbeat, page-load smoke, resource alerts, migration timing as App Insights event).

## Surprise

- **Both slots looked dead for ~6 minutes after `deploy.ps1` finished cleanly.** Drew ran `deploy.ps1`; script output was perfect, both grant lines printed (`Granting prod identity on 'booktracker'...` / `Granting staging identity on 'booktracker-staging'...`), no errors. He restarted the staging slot to trigger migrate-on-startup. Both URLs started returning 500s. The custom domain `books.silly.ninja` came up with a generic ASP.NET error page. Prod-slot diagnostic logs showed `Microsoft.Data.SqlClient.SqlException ... A connection was successfully established with the server, but then an error occurred during the login process. (provider: TCP Provider, error: 35 - An internal exception was caught) ---> System.IO.IOException: ... Connection reset by peer.` Recovered ~60 seconds after a full Stop/Start (not just Restart) of both slots.
- **The error signature is specific: TCP succeeded, AAD login failed.** "Connection successfully established with the server, but then an error occurred during the login process" + "Connection reset by peer" is the Azure SQL signature for AAD authentication being rejected at the protocol level — usually because the managed identity's user mapping in `sys.database_principals` isn't valid yet, or because the AAD token presented isn't accepted by the server. Different from "Login failed for user X" (SQL-auth signature) or "A network-related or instance-specific error occurred" (network signature). Worth knowing the distinction — narrows the diagnosis from "all the things" to "auth state" in seconds.
- **The Bicep redeploy stacked enough state-changing operations to briefly destabilise auth.** Best guess at root cause: redeploying the SQL server `administrators` block (even with unchanged values) re-applies AAD admin config and resets active connections; `publicNetworkAccess` toggled Disabled → Enabled → Disabled within ~30 seconds for the firewall pinhole; the App Service slot worker's AAD-token cache lagged the new identity grants; first migrate-on-startup against the empty staging DB held the worker process briefly. Each is benign in isolation; together they need a window of seconds-to-minutes to settle. A `Restart` of the slot worker isn't enough — it recycles the worker process but reuses the underlying VM/sandbox state. `Stop` + `Start` re-allocates the sandbox and forces a fresh token acquisition, which is what fixed it.
- **Slot-sticky CS was the part that worked, not the part that broke.** Easy to misdiagnose this as "the slot-sticky change broke prod." The prod slot's `DefaultConnection` never moved off `Database=booktracker` — verifiable from the failing log line itself ("connection to database 'booktracker'"). The lag was in the auth layer downstream of the CS, not in the CS routing. If we'd misread the diagnosis we'd have torn out the slot-sticky pin (the actual safety feature) instead of doing Stop/Start. The discipline that saved us: always read the failing log line for ground truth on where the connection is going *before* assuming the most-recently-changed thing broke.
- **Drew's heart-palpitation moment is real even on a single-user app.** This is "a few hundred books captured for my own usage" — no other users, no SLA, no on-call rotation. And yet the gap between "site is up" and "site is down" was qualitatively different from any local-dev failure. Drew's exact framing: *"a single user, few hundred books captured for my own usage and I still had a heart palpitation moment when I thought I had lost it all. And I knew it should work cause I reviewed the code, but production being down is... not ideal."* That's the load-bearing observation: code review proves the *code* is right; it doesn't prove the *deploy* will be. Solo-dev infra is real infra the moment the data is real.

## Lesson

- **`Stop` + `Start` is a different operation from `Restart` for App Service slots.** `Restart` recycles the worker process; `Stop` + `Start` re-allocates the underlying sandbox/VM. For auth-cache or token issues the latter is what you want — it's the difference between "process restart" and "fresh process on fresh sandbox." Worth knowing the distinction *before* you need it. The runbook line: "if both slots are unhealthy after a deploy, try Stop/Start before declaring failure."
- **Bicep redeploys can stack transient failures even when nothing functional changed.** SQL admin block re-application + public-access toggle + slot config update + first migration is a lot of state-changing operations in a small window; each is individually benign and they collectively cause auth-state lag. Mitigations: do the riskier deploys at low-traffic times; expect a ~5min settling window before declaring failure; have Stop/Start in your back pocket. Long-term: prefer narrower Bicep changes that don't re-touch the SQL admin block, and consider explicit `dependsOn` chains that serialise state-changing operations rather than letting ARM apply them in parallel.
- **Empty staging catches schema syntax errors and misses every data-shape failure.** The deploy-safety value of separating staging from prod comes in two parts: (a) schema can break without taking prod down — solved by the separation itself; (b) staging actually catches the failure modes that matter — solved only by giving staging representative data. NOT NULL against existing nulls, unique-index violations against existing duplicates, FK orphans, CHECK violations, type-conversion failures, performance-on-real-data — none of these surface against an empty DB. EF migrations on SQL Server are transactional, so prod can't *corrupt* — but the deploy still fails after the PR has merged. The next TODO (#1, bacpac sync) closes that gap.
- **The "review the code" gap.** Code review catches code bugs. It does not validate deploy-stack timing under specific orchestration sequences. The 6-minute outage didn't come from a code defect — it came from the order in which ARM applies resource changes interacting with App Service worker auth-cache state. No amount of re-reading the Bicep would have predicted it. Mitigations are operational, not in-code: synthetic monitoring (filed as TODO #12), a known recovery procedure (Stop/Start), and the willingness to wait 5–10 minutes before declaring failure on a fresh deploy.
- **Lift the recovery procedure into the runbook the moment you discover it.** `infra/README.md` already had the order-of-operations note (run deploy.ps1 → restart slot → verify) but the "if both slots are down post-deploy, try Stop/Start before panicking" line was still in conversation. Worth adding to the README on the next pass. The general pattern: a recovery procedure that's been used once successfully is worth ~30 seconds of doc time so the next time it happens (or someone else hits it) it's a step on a checklist instead of a 6-minute diagnostic loop.
- **Read the failing log line before assuming the most-recently-changed thing broke.** When something goes wrong right after a deploy, the temptation is to revert the change. But the failure mode often isn't in the change itself — it's in something the change interacted with. Reading the log line gives you ground truth ("prod is hitting `Database=booktracker`, the right DB" — so CS routing is fine) and prevents you from breaking a working safety feature in the name of fixing a different problem.
