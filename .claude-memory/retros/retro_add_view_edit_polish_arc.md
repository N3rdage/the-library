## Shipped

Six PRs merged, 2026-05-10 → 2026-05-13:

- **PR 1 (#239)** — Status chip picker on View was visually broken (clicks didn't shift colour because `Color` was unconditional). Fix: `Variant` + `Color` computed per-chip from selection state — Filled+semantic on selected, Outlined+Default on others. Initial fix used `Color.Default` for Unread; Drew's "VERY subtle on my monitor" tweak landed `Color.Warning` (brass-gold) and bold weight via `Style`.
- **PR 2 (#240)** — Genre picker unification. Replaced the Bootstrap `GenrePicker` with `MudGenrePicker` on Add Book and the Work edit dialog so chips + typeahead behave the same on every surface. Dropped the "Browse all Genres" expander (the tree view nobody used). Per-Work scope labelling on `WorkEditDialog`.
- **PR 3 (#241)** — Collection-mode Genre toggle mirroring the just-shipped Single-Author pattern. "Same genre(s) for all works" → shared picker; OFF → per-row pickers with inheritance on `AddCollectionWorkRow`. Bidirectional toggle propagation (OFF→ON = union of per-row to shared; ON→OFF = broadcast shared to every row) so flicking doesn't lose picks. Drew's driver: "20 stories, 19 SF and one outlier — capturing genres once instead of 20 post-save edits is an order of magnitude better."
- **PR 4 (#242)** — `AddExistingWorkDialog` on Book Detail. `IWorkSearchService` typeahead with "Title (year) — Author · in N books" disambiguation. `excludeBookId` filters out already-attached Works at the query level; defensive double-attach guard handles stale dialog state. Surfaced **the Blazor dialog VM lifetime trap** — `@inject BookDetailViewModel` inside the dialog handed a fresh Transient instance (whose `Book` was null), silently breaking search. Fix: pass the page's initialised VM via `DialogParameters`, matching the existing `EditionCoverUploadDialog` pattern.
- **PR 5 (#244)** — Keyboard-first existing-Work autocomplete on Add Book collection rows. Each row's title field becomes a `MudAutocomplete<WorkSearchResult>`; type → existing matches surface; Down + Enter attaches the highlighted one (row flips to a compact summary card); free-text Enter passes through to "add new row + focus next" (the existing PR 5-prior pattern). `SingleAuthor` / `SingleGenre` toggles skip attach rows. Collection toggle confirms via `ShowMessageBoxAsync` when any row is in attach mode — fat-finger flick recoverable. **The hard part**: `collection-works.js` had to suppress MudAutocomplete's "commit first-highlighted item on Enter" without breaking the navigated-pick case. Solution: capture-phase keydown listener tracks navigation per input via a WeakSet (Down/Up sets the flag; printable keys / Backspace / Esc clears it); on Enter, defer to MudAutocomplete only when the flag is set, otherwise `stopImmediatePropagation` to block the auto-pick.
- **PR 6 (#245)** — `/edit` decommissioned. `AddWorkDialog` on View combines attach-existing + create-new under one entry point ("Add, but allow me to select existing if one exists" — Drew's framing). Same JS pattern as PR 5's row autocomplete, reused via a sentinel index. Delete Book affordance with confirm. Remove Work from Book (detach if has other Books, delete outright if orphan). Rewired six surfaces that linked to `/books/{id}/edit`. Net delete: ~960 LOC (`Edit.razor` + `BookEditViewModel` + the superseded `AddExistingWorkDialog`).

Pre-arc state: the View page (`/books/{id}`) and the Edit page (`/books/{id}/edit`) were doing overlapping jobs — View had a "Full edit page" escape hatch into Edit, Edit had a "Back to view" link going the other way. The arc removed the escape hatches by making View competent enough to be the only edit surface.

## Surprise

- **MudAutocomplete auto-highlights the first match — there's no off switch.** PR 5 went out treating "if `aria-activedescendant` is set, defer to MudAutocomplete on Enter" as the right deference rule. Drew's repro killed it: type "Con" → first match "Condor" auto-highlighted → Enter commits "Condor" instead of capturing "Con". The aria attribute fires the moment the dropdown opens, regardless of user navigation. Fix needed real navigation tracking (capture-phase keydown listener watching Down/Up vs printable keys), not just an aria check. The MudBlazor 9.4 XML docs have no `AutoHighlight` / `SelectFirstByDefault` property — verified by greping the XML doc file. Either we override it in JS or we live with the bug.
- **`input.blur()` doesn't close a portaled popover; nor does dispatching a synthetic Esc; nor does removing `.mud-popover-open` directly.** Three attempts to close MudAutocomplete's dropdown after our `stopImmediatePropagation`'d free-text Enter, all failed. The popover is portaled to `MudPopoverProvider` at the page root, so it doesn't share a parent with the input — focus loss doesn't cascade. Synthetic `KeyboardEvent` dispatches don't reach Blazor-delegated event handlers because Blazor's event delegation appears to filter `isTrusted=false`. Removing the open-class directly drifted MudAutocomplete's internal `_isOpen` vs DOM without actually hiding the popover (suspect transform/opacity is doing the hide, controlled by `mudpopoverHelper`). After three attempts Drew said "let's just go with escape." Inline note in the JS prevents the next reader re-walking the dead ends.
- **`ShowMessageBox` was renamed to `ShowMessageBoxAsync` in MudBlazor 9.4.** Initial compile error: "no extension method `ShowMessageBox` accepting first argument of type `IDialogService`." Spent ~3 minutes diagnosing before grepping the package XML doc. Worth knowing because the LLM training data still has the old name everywhere.
- **`Add.razor`'s constructor injection silently broke `BookAddViewModelTests` when I added `IWorkSearchService` to the VM.** Build error surfaced it within seconds, but the lesson is broader: every test factory method that constructs a VM by hand is a coupling point. Add a dep to the VM, update the factory. There's no abstraction protecting against this — and a service locator pattern would be worse. Just a discipline thing: when you change a constructor, grep for `new <VMName>(`.
- **PR 6 audit had one factual error in the agent's report.** The Explore subagent's audit said "Genres editing — Currently read-only on View" and flagged it as a gap. But `WorkEditDialog` (opened from "Edit work" on a single-work View page) has had `MudGenrePicker` since PR 2 of the arc. The subagent had seen the *inline display* (read-only chips on the main panel) and conflated that with "no edit path." Lesson: subagent audits are good for inventorying surfaces but should be verified against the actual edit flow before scoping work off them. Cost was small here (I noticed before writing the wrong code) but easy to imagine a worse version where the audit pushed me into building a redundant dialog.
- **The /edit page deleted in 386 lines, the VM in 574, and nothing else broke.** The PR's net was +466 / −1182 across 17 files, and the test suite turned green first try. The decoupling work was done across PRs 1–5 — by the time PR 6 ran, the only thing still pointing at /edit was navigation, not feature dependency. **Lesson — generalises:** if a "delete this surface" PR is large because of the *delete*, it's usually fine. If it's large because of the *re-port*, the earlier PRs missed a tax.

## Lesson

- **"Ask, but allow free-text" is the recurring UX pattern.** Drew steered me to this shape twice in the same arc: in PR 5 (collection row title) and in PR 6 (View-page Add Work). Both times the first proposal was "two buttons" or "two-stage dialog with a toggle." Both times Drew's correction was the same: *one entry point, type to discover existing, fall through to capture new*. The principle: the user shouldn't have to know in advance whether what they want exists. The cost of "always try the search first" is zero keystrokes (debounced, server-side); the cost of guessing wrong on the toggle is broken flow. This pattern is worth a memory of its own (`feedback_add_with_optional_existing.md`).
- **MudAutocomplete + capture-phase JS is the recipe for the auto-pick problem.** Three things together: (a) `CoerceText="false"` so the typed text survives Esc / un-picked close; (b) capture-phase keydown listener that fires BEFORE MudAutocomplete's; (c) per-input WeakSet tracking whether the user has navigated via Down/Up since the last text edit. Free-text Enter `stopImmediatePropagation`s; navigated Enter passes through. Reusable across any inline-search-with-fallback surface — captured as its own memory entry.
- **Stop fighting after three attempts; document the workaround inline.** The popover-stays-open ordeal in PR 5 burned ~30 minutes across three commits before Drew said "let's just go with escape." The retrospective question: was there a tell at the start? Yes — when the symptom was "MudBlazor's portaled popover doesn't close on focus loss," the implication was MudAutocomplete's internal state machine owns the open-flag, not the DOM. Once attempt 1 (`blur()`) and attempt 2 (synthetic Esc) had both failed *through different mechanisms*, the third attempt (direct DOM class manipulation) should have surfaced "I don't have a way in." The inline JS comment now lists all three dead ends so the next reader doesn't re-walk them. **Generalises:** when a fix has failed twice via different mechanisms, the third attempt is usually fragile; accept the workaround and write the note.
- **The Blazor dialog VM Transient trap is recurring enough to memoise.** Already captured (`feedback_dialog_vm_lifetime.md`), but worth restating in context: the symptom was silent ("no Works found" when the search service was actually working fine — the dialog's VM just had `Book = null`). The fix is two characters of XML in the existing pattern: `DialogParameters<TheDialog> { { x => x.VM, VM } }` and a `[Parameter]` declaration on the dialog instead of `@inject`. The trap is that every existing dialog without shared state works fine via `@inject`, so the failure mode is invisible until a dialog needs the page's initialised state. **Generalises:** Blazor Server's "every render is a circuit roundtrip" and Transient lifetime are both right defaults — they compose badly when a dialog needs hot state. Reach for cascading or `DialogParameters`.
- **Close TODOs in the same PR as the work that delivers them.** Drew's standing rule going forward (set 2026-05-13): the row moves to Shipped in the same commit that ships the code. Avoids the slow drift where Shipped diverges from reality. Memorialised as a feedback memory.
- **A six-PR arc with strict layering kept each PR mergeable.** The arc could have been "the View page rework PR" — one giant diff. Instead it was six PRs each shippable on its own day. The constraint forced ports to happen before deletes (PR 4 dialog before PR 6 page removal), and surfaced the dialog VM trap on PR 4 in a small enough context to fix cleanly. **Generalises:** when you're decommissioning a surface, ship the replacement first as additive work, then the delete is a chore. The mobile-arc retro lesson ("ship the working core, then ship polish as its own PR") applies — same shape.
