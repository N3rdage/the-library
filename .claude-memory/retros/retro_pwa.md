## Shipped

PR #91 — 11 files, +225/-2. BookTracker is now installable as a PWA on iOS Safari, Android Chrome, and desktop Edge/Chrome/Firefox. Launches in standalone mode (no browser chrome), home-screen icon, theme colour integration with mobile status bars. Network-first-with-cache-fallback service worker caches static assets for faster repeat loads; `/_blazor/*` SignalR traffic passes through untouched.

## Surprise

- **Generating PNG icons without tool dependencies is easier than expected.** Initial instinct was "user needs to install ImageMagick or Inkscape to convert the SVG master". But `System.Drawing.Common` + `Add-Type -AssemblyName System.Drawing` in PowerShell lets you draw a 512x512 book icon in ~40 lines of imperative GDI+. The SVG still lives in the repo as source-of-truth for the *design*; the script redraws the same design via GDI+ for the *raster assets*. Reproducible, repo-contained, no external tool install. Only works on Windows (System.Drawing.Common) but this is a Windows-only dev project anyway per CLAUDE.md.
- **PWA icon URL split: stable URL for the service worker, versioned URL for the bootstrap.** Blazor's `MapStaticAssets()` fingerprints static assets via `@Assets[...]` for cache-busting — perfect for CSS, JS, etc. But service workers need stable unversioned URLs so the browser can detect updates. Resolution: reference `/service-worker.js` directly (stable URL, raw static file); reference the registration stub `service-worker-register.js` via `@Assets[...]` so the bootstrap itself cache-busts per deploy but ships pointing at the unversioned SW path. Subtle but load-bearing — got it right first try by sanity-checking the Blazor docs.
- **"Install PWA" support is silently better than I expected.** I'd braced for some mobile-Safari-specific bug or missing manifest field. None. Standard manifest.webmanifest with an icon array including a `maskable` purpose variant just works across Chrome/Edge/Safari. iOS needed the `apple-mobile-web-app-*` meta tags, but those are well-documented and stable. The whole PR ended up smaller than the Trove lookup PR.
- **Interactive Server vs Blazor WASM PWA scope mismatch** — needed flagging but wasn't a blocker. The "PWA" template in the Blazor WASM world implies true offline. Our Interactive Server app needs SignalR, so "offline" isn't realistic. The delivered scope is "installable + standalone + cached assets for perceived-perf" — absolutely fine for Drew's phone-side barcode-scanning flow (always online). Worth calling out in ARCHITECTURE + commit message so future-me doesn't get confused.

## Lesson

- **"What do I need an external tool for?" is often the wrong question.** First instinct with SVG-to-PNG was "install ImageMagick/Inkscape and document it". Second instinct was "embed SVG in the manifest and hope browsers do PNG conversion" (unreliable for iOS). Third instinct was the answer: **System.Drawing can draw**. The icon design doesn't need to be pixel-identical to the SVG — they share the same mental model (purple background, darker spine on left, three white text lines) but are independent rasterisations. Repo stays clean, Drew doesn't need new tooling, PNGs regenerate from a script.
- **Stable vs versioned URLs is a real load-bearing distinction in modern SPAs.** Blazor's `@Assets[]` system and service-worker URL requirements are in direct tension. Getting this wrong means either (a) SW never updates after first install or (b) every deploy breaks the SW registration. The pattern "stable URL for the thing with its own update-detection mechanism, versioned URL for the thing that just needs cache-busting" generalises beyond SW — same logic applies to OAuth redirect URIs, webhook URLs, deep links. Worth remembering.
- **Pragmatic PWA scope for server-rendered apps.** "Make it installable + standalone + a bit snappier on repeat loads" is 90% of the benefit of a PWA for most server-rendered apps. True offline is the remaining 10% and costs 10× more to build. Drew gets what he actually asked for ("on mobile would be better if we could run the website as if it was an App") without trying to convert the app to something it fundamentally isn't.
- **Binary assets in repos: how much ceremony is right?** I committed the generated PNGs alongside the generator script. Alternative: gitignore the PNGs and regenerate at deploy/build time. Chose committing because (a) the icons change rarely and (b) having them in the repo means `git clone → dotnet run` works without first running the PowerShell script. The tradeoff is real binary-in-git bytes; acceptable at ~7KB combined. Flag for later: if icon sizes grow or start changing often, consider moving to a build-time step.

## Quotable

"How do I turn an SVG into a PNG without any external tools?" is a question that used to take 15 minutes of Stack Overflow reading plus a tool install. Now it's `Add-Type -AssemblyName System.Drawing` and 40 lines of GDI+ primitives that redraw the same design imperatively. The SVG still lives in the repo for the humans reading the design; the PNGs live in the repo for the browsers reading the manifest. Both are generated from the same mental model, both are reproducible from the checkout. One less tool to install, one less decision about whether to commit binaries. The repo is the single source of truth for *everything* the app needs to run.
