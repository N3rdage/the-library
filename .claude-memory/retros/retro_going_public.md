## Shipped

## Surprise

- **Drew picked Option A, not my recommended B.** I'd recommended untracking `user_drew.md` on the grounds that it reads personal (describes collaboration style). Drew said keep everything — because *this project is an experiment in AI-first development* and the workflow-preference files are its most distinctive artefact. The reframe made me rewrite the README: instead of "book tracker + some docs" with a contrib-policy footnote, the README now leads with "nearly all code is Claude-authored, here's what's interesting about that, here's the contribution policy because the experiment's frame doesn't accept external PRs." That's a bigger structural change than "keep one more file tracked" and I wouldn't have made it if I'd just done Option B by default. **Lesson: planning docs that ask questions and wait have a real information-extraction function beyond choose-your-own-adventure; the answer can change the work.**
- **"Experiment framing" vs "product framing" in the README is a substantive choice, not a stylistic one.** My first instinct was a product-forward README: "BookTracker is a Blazor Server app for cataloging your personal library..." with the AI-collaboration angle as a bullet point near the bottom. That framing is *technically accurate* and completely wastes what's actually interesting about the repo to a stranger landing on it. The version that shipped leads with the experiment — "Nearly all code is written by Claude Code" — and the product description becomes the second paragraph. Visitors who care about "Blazor Server demo" find what they need; visitors who care about "how does that actually work" get the hook first. Applies past this project: when a repo's distinguishing feature is meta-level (how it was built, what methodology it explores), the README should lead with the meta, not the product.
- **Going-public anxiety is mostly about the wrong things.** The scary-feeling question is usually "did I leave credentials somewhere?" — which for this repo was answered definitively by the security audit + history secret scan. The actually-substantive question turns out to be "what part of the development context am I okay being indexed by search engines and crawlers alongside my name?" — which is the `.claude-memory/` question. Credentials are a yes/no; personal disclosure is a graduated preference. The planning doc's structure (three lettered options) made that preference explicit; the default "make public and hope" wouldn't have surfaced it.
- **GitHub's Code security page was different on the private repo vs what I expected.** When Drew walked through the settings, several features I'd listed (Secret scanning, Push protection, CodeQL, Private vulnerability reporting) weren't in the UI at all. The reason: those are free on public repos but require paid GitHub Advanced Security on private ones, so GitHub just hides them from the UI to avoid "click this → $$ prompt." Once Drew flipped, they appeared. **Durable rule: "what's in GitHub's Settings UI" is a function of the repo's visibility + subscription tier, not a fixed list; when writing guidance, note the tier + visibility assumption, not just the feature name.**

## Lesson

- **Planning docs with explicit options uncover user preferences that silent defaults would miss.** If the `GOING-PUBLIC.md` doc had been "here's what I'm about to do" with no choice presented, Drew would have got Option B silently and the README would have been a product-first bullet-point. The three-option structure surfaced "actually the experiment angle is the point" and led to a different README + a different contribution policy. Generalises: when a decision has multiple defensible paths with user-preference-dependent tradeoffs, enumerate them and ask. Skip this when there's truly one right answer; apply it when there's a judgment call about identity, privacy, audience, or scope.
- **Lead with what makes the repo distinctive, not what makes it legible.** Product-forward README is the safe, legible default — copy-paste-adaptable from any template. Meta-forward README requires actually knowing what's interesting about this specific project. The cost is writing a README that's unique to this repo; the payoff is a first-visitor experience that demonstrates the project's actual value proposition in the first paragraph. For a portfolio project or an experiment-driven repo, the meta lead is right; for a product-driven repo where the meta is incidental, the product lead is right. Know which kind of repo you have.
- **"What would a stranger see?" is a different scrub than "what's a security risk?"** The security audit answered the latter: no credentials, no secrets, nothing exploitable. The going-public review added the former: some identifiers (subscription naming conventions, domain name, workflow-preference files) aren't *risks* but *disclosures*. Most disclosures are fine for a personal / portfolio project; a few are preferences. Worth a separate pass with that framing, not folded into security.
- **Post-flip TODO rebalancing is part of the flip, not afterthought.** The flip unblocks several things (`Environments` availability, public Dependabot features, Code scanning, free Actions minutes), blocks nothing new, and changes the shape of one row (reviewer-approval → wait-timer). Rebalancing the priority list reflects the new state; without it, the list silently rots. Same shape applies to any "major gate flipped" event — ship, deploy, go-to-market, open-source — each of them reshuffles what's blocked vs actionable.

## Quotable

A planning doc that says "here's what I'm going to do" is a status update. A planning doc that says "here are three options with tradeoffs, my recommendation is B, but pick" is a structured ask. The first is efficient when the decision is obvious; the second is load-bearing when user preference changes the downstream scope. For this repo, the `.claude-memory/user_drew.md` question looked like a file-tracking toggle and turned out to be "what kind of repo is this?" — Drew's answer (this is an experiment, the workflow artefacts are the point) reframed the README from a product page to an experiment page. The three-option structure made that reframe visible; a silent default would have taken the safe-and-wrong path. **The question asked upstream changes the work done downstream.** Plan with that in mind.
