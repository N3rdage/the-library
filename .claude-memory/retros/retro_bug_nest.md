## Shipped

Three fix PRs in one session:
- **#103 — Dialog double-click guard.** Fast double-tap on any of Detail.razor's eight dialog-trigger buttons opened two stacked `DialogService.ShowAsync` instances. Fix: single `dialogOpening` flag + `OpenDialogGuardedAsync` helper wrapping all eight openers, with `Disabled="@dialogOpening"` on each MudButton.
- **#104 — AI config slot-sticky + always-emit KV refs.** Anthropic key vanished on repeat slot swaps. Root cause: `app-config.bicep` conditionally omitted the `AI__Anthropic__ApiKey` app setting when `-AnthropicApiKey` wasn't supplied on a deploy, AND no settings were slot-sticky. Fix: always emit every KV reference (secret is conditional in KV; the *reference* is unconditional in app settings) + added a `slotConfigNames` resource marking AI/auth settings as sticky.
- **#105 — Shopping page NRE on ISBN match.** `SearchByIsbnAsync` and `SelectBookAsync` loaded `Book.Works` without `.ThenInclude(w => w.Author)`, so `PrimaryAuthor` dereferenced a null navigation. `SearchByTextAsync` (typing a title) had the right Include — which is why text search worked and ISBN crashed. Fix: add the Includes + harden `PrimaryAuthor` with null-tolerance.

## Surprise

- **The scanner "bug" on Shopping was a red herring.** Drew reported "scanner not working on the Shopping page." I compared Shopping and BulkAdd scanner code side-by-side — structurally identical. Couldn't reproduce from here, asked for more info. Turned out the scanner was fine; what *was* broken was the ISBN-found path (whether the ISBN arrived via scanner OR manual typing). The scan succeeded, then `SearchByIsbnAsync` crashed silently mid-handler, and the UI just sat there with the scan window closing. The surprise isn't the NRE — it's that "scanner doesn't work" and "any ISBN lookup with a match crashes" look like the same bug from the user's POV. **Lesson: when a user reports a UI feature "not working," ask whether the underlying action completes. If the scanner's only job is to hand an ISBN to a VM method, and that VM method throws, the scanner looks broken even though it did its job.**
- **All three bugs had a pattern already in the codebase that would have caught them.** 
  - For #103, dialog handlers with `saving` flags existed (every one of my PR 3/PR 4 dialogs uses one). The *trigger* buttons needed the same discipline as the *save* buttons inside the dialogs — I just didn't think to apply it there.
  - For #104, the "always emit stable references; gate the underlying secret separately" pattern is literally how OpenAI is set up in the same Bicep file (`AI__AzureOpenAI__ApiKey` is always emitted; the key secret is written unconditionally by `ai-services.bicep`). Anthropic and Trove were written with a conditional app setting for reasons that made sense in isolation but drifted from the project's default pattern.
  - For #105, other VM queries in this project correctly Include every navigation they dereference (e.g. `SearchByTextAsync` in the same file). The two broken methods were outliers — and because the NRE only surfaced on ISBN match (a path no test exercised before the fix), it shipped quietly.
- **The shopping page had no tests until this session.** Three new tests shipped with #105. Without them, the include-gap was invisible to CI. The VM had been touched multiple times in its lifetime without anyone noticing two of its three query methods were gonna NRE on a cache-warm load. "No tests" correlates strongly with "latent bugs that ship" — the missing coverage wasn't a stylistic choice, it was an absence of pressure to surface the problem.

## Lesson

- **Apply project patterns consistently, even for adjacent code you're not actively touching.** All three bugs came from local code that didn't follow the project's own established pattern elsewhere. Cost to apply the pattern was small; cost to discover the bug was high (user-visible NRE in prod, deploy-time config drift, UX glitch). A useful mental checklist for PR review: "Does this file handle X the same way the rest of the project does?" If not, is there a specific reason, or is it drift?
- **"Always emit the reference, gate the secret"** is the pattern for KV-backed app settings and worth stealing broadly. Conditional app settings are fragile because the app-settings resource in Bicep is replace-all. If a setting is sometimes-present, any deploy that passes the absent branch strips it — which is usually wrong. The robust shape: **every reference always present, every secret independently gated, slot-sticky list covers anything that should survive swap-plus-redeploy**. Works for Azure Key Vault references; generalises to any "settings that point at secrets" model.
- **Double-click guard is 6 lines per handler-cluster, not per handler.** One shared `bool dialogOpening` + one `OpenDialogGuardedAsync(Func<Task>)` helper + `Disabled` binding on each trigger. Eight buttons on Detail.razor pay for the helper in one pass. Worth being the default shape for any page with multiple async-button triggers — the alternative (per-handler `if busy return` scattered across eight methods) is both more code and harder to see at a glance.
- **Regression tests are cheap when you've just fixed the bug.** The three tests on `ShoppingViewModelTests.cs` (seed book, call VM method, assert author populated) took 10 minutes to write. They would have caught the original bug in 1 CI cycle. The decision of "do we write a regression test for this?" shouldn't be on the critical path of shipping a fix — it should be reflex. The test seed patterns (`TestDbContextFactory`, `db.Books.Add(new Book { ... })`) are already well-worn in this repo; there's almost no friction to adding a test for any VM method.
- **"Scanner doesn't work" might not be about the scanner.** When a user reports a UI surface failing, broaden the diagnosis to "does the whole action complete?" — especially when the surface's job is to hand input to a downstream handler. The scanner's only failure mode of "finds a book → closes window → shows nothing" is a giveaway that the downstream handler threw and ate the error. Lesson generalises: UI "not working" reports are often failures a step or two past the UI.

## Quotable

All three bugs had pre-existing patterns in the codebase that would have prevented them. The dialog `saving` flag was already everywhere inside the dialogs — just not on the buttons that opened them. The "always emit KV reference" shape was how OpenAI was set up — just not how Anthropic and Trove were. The `.ThenInclude(w => w.Author)` was in `SearchByTextAsync` — just not in the other two methods in the same file. The durable lesson: **the time to apply a pattern consistently is when you first introduce it, not when you notice the second file also needs it.** Pattern drift across adjacent code is a bug-shaped gap that accrues silently and only surfaces when someone walks the specific path. The prevention is not more discipline on future changes — it's a pass through adjacent code at pattern-introduction time. "I just added X pattern here; what other places in this file/project should use it?" is a short checklist that pays for itself in the first bug it prevents.
