## Shipped

Three code PRs, 2026-06-30 → 2026-07-01, expand→cutover→contract. The trigger was a one-line dogfooding realisation, not a planned feature.

- **PR1 (#401) — expand + back-fill.** Added `Book.SeriesId/SeriesOrder/SeriesOrderDisplay` + aggregate methods + `Series.Books` nav, with a migration that back-fills the Book columns from each book's Works. **Work columns retained** as the live source — pure groundwork, no behaviour change. A read-only DB check first proved the back-fill was 1:1 (71 series-works → 71 distinct books, 0 conflicts); verified 71/71, 0 mismatches after apply.
- **PR2 (#402) — the cutover (atomic).** Flipped every reader and writer Work→Book in one PR: writers (Add/Bulk-Add assign the Book; series picker moved from the per-Work edit dialog to the **Book** edit dialog; `UpdateBookDetails` carries series with **required** args; `MergeWorks` drops series; the series-management page manages Books); readers (gap detection web + AI context iterate `s.Books`, converging with what mobile already did; the snapshot projects straight off the Book, deleting the lossy first-Work hack; book detail / series list / library filter+sort / author detail + rollups all read book-grain). **Two review rounds** before merge (see Lesson).
- **PR3 — contract.** Dropped `Work.SeriesId/SeriesOrder/SeriesOrderDisplay` + the Work aggregate's `AssignToSeries/ClearSeries/SetSeriesOrder` + `Series.Works`; migration with a **back-filling Down** (restores Work series from Book, symmetric to PR1). Carried the doc updates (CLAUDE.md, ARCHITECTURE.md, TODO #56→Shipped) + this retro. Spawned **TD-20** (the duplicated SharedParsers-vs-GetSeriesGaps gap computation, deferred).

Drew's call at the fork (2026-06-30): **Option A** (move series to the Book) over "don't split collections into Works" or "support both axes"; series editing moves to the existing Book edit dialog.

## Surprise

- **The model was already half-migrated — and had a latent bug on exactly this case.** The PR1 mapping found the mobile cache (`CachedBook`) and the wire format (`BookSnapshot.SeriesId`) *already* stored series at book grain; the web bridged to them by projecting a book's series from its **first Work by Id** (`GetCatalogSnapshot`), explicitly commented "Series membership lives on Work… take the first Work." For a single-Work book that's lossless; for a multi-Work collection it shipped the *first story's* series (usually null) — the exact failure Drew felt in the app. So the cutover **deleted a workaround** instead of building new machinery, and a whole consumer (mobile) was "free" — it just started receiving correct data. A downstream consumer's workaround is a signpost to where the model wants to go.
- **The contract column-drop was a free reference-finder — and caught a PR2 miss.** Dropping `Work.Series*` turned "did I update every reader?" into a compile error list: the `BookTracker.Tools.SnapshotDump` diagnostic tool (not in the 6-project mental model), dead `SeedWorkAsync` test helpers, and — tellingly — a `DeleteSeries_…SetNullsMemberWorks` test I'd **missed converting in PR2** (the FK moved to Book, so the SET NULL now clears `Book.SeriesId`, but the test still asserted `Work.SeriesId`). It compiled and passed in PR2 because Work still had the column; the contract made it surface. Expand/contract's third phase is a correctness net, not just cleanup.
- **A grep that LISTED the buggy files didn't stop me shipping the bugs.** In PR2 my own grep returned `AIAssistantViewModel`, `AIAssistantService` (×3 providers), and `SeriesMatchService` as files containing `SeriesId` — and I acted on `SharedParsers` only. Round-1 review found 3 silent-failure bugs in exactly those listed-but-untouched files (AI "create collection" wrote the Work → no-op'd; AI "suggest collections" read stale Work series; series-match Strategy 2 counted Work series → spurious hint).

## Lesson

- **A grep's file list is a checklist — tick every entry or record why it's skipped.** The single most expensive miss of the arc was treating a grep result as "found the main one" rather than "here are all N sites." Three confirmed correctness bugs (one triplicated across providers) all sat in files my own PR2 grep had surfaced. The discipline isn't "grep harder" — it's, after the grep, enumerate each hit and classify it (fix / skip-because / false-positive) before moving on. The cutover's "every reader and writer flips together or new writes diverge across two columns" makes a *missed* reader especially nasty: it fails silently, not loudly. → candidate feedback memory [[feedback_grep_list_is_a_checklist]].
- **Default args on a write command that reconciles state are a silent-clear footgun.** `UpdateBookDetails` gaining `SeriesId/SeriesOrder/SeriesOrderDisplay` — I first defaulted them to null for terse callers, then caught that a save *always reconciles* membership, so any 4-arg call would silently wipe a book's series. Made them **required** so every caller is explicit. The handler can't distinguish "omitted → leave alone" from "null → clear" through a defaulted param; if a command reconciles, its inputs aren't optional. Same family as [[feedback_overloaded_field_shared_predicate]] — the danger is an invisible second meaning at the call site.
- **Re-review after applying fixes; two rounds converge and tell you when to stop.** Drew asked for a review before push (round-1: 3 real silent-failure bugs) and then "again just to be safe" (round-2: 2 minor + cleanups, no severe). Round-1 = real bugs, round-2 = polish is the convergence signal — a third pass wasn't needed. Round-2 also *refuted* a finder's EF-translation scare on the new author subquery: an independent verifier found the exact shape already ships and works in `DuplicateDetectionService`. Verification kills false positives as well as confirming real ones; I still added a real-SQL test to settle it for good. Both rounds presented findings → **waited** for Drew's cut before fixing ([[feedback_review_findings_gate]]).
- **An atomic cutover can't be sub-split, but expand/contract still buys PR-level safety.** Readers and writers had to flip in one PR (PR2) — split them and a freshly-edited book diverges between `Work.SeriesId` and `Book.SeriesId`. What the 3-PR shape bought was *deploy* safety: PR1 lands + verifies the data move before anything depends on it; PR3's drop is reversible (back-filling Down) and is where the destructive migration's review-tag lives. Don't conflate "the behaviour change is atomic" with "the whole arc must be one PR."
- **A model move drags its editing UI with it.** Series lived in the per-Work edit dialog only because it lived on the Work. Moving membership to the Book meant the picker had to move to the Book edit dialog — a real UX consequence, not just a data change. Flagged to Drew as the one genuine fork (where do you edit a book's series now?) rather than guessed.

## Known limitations (recorded, not fixed)

- **TD-20 — the gap computation is duplicated** between `SharedParsers.BuildLibraryContextAsync` (AI prompt) and `Wishlist/GetSeriesGaps`. The cutover had to edit both copies identically; the shared *predicate* (`SeriesSlots.OccupiesNumberedSlot`) is already factored, the per-series owned/missing math around it isn't. Low drift risk (single user), deferred.
- **A work's series on the author-detail page is derived from its book(s)** (lowest SeriesOrder wins for a Work spanning multiple books/series). 1:1 for ordinary single-Work books; the multi-book-different-series case is rare and picks one deterministically.
- **`Detail.razor` series-label formatting is duplicated** across the single-Work and multi-Work branches (mutually exclusive, low drift) — Drew's call to leave it.
