## Shipped

PR #107 — `/authors` rebuilt in MudBlazor (replacing a Bootstrap table), each row expandable with a per-row toggle between **Works** and **Books** views. Canonical rows roll up works/books under their aliases; alias rows show only their own. Author-authored-under-alias rows carry an "as Richard Bachman" chip so the attribution is visible inside the rollup. Rename + alias-management actions migrated to MudBlazor equivalents (MudButton / MudSelect) and continue to work whether the row is expanded or not.

Deep link: `/authors?expand=<id>` pre-opens that author's drill-down. Wired into Home's Top-10 author list — clicking a bar jumps straight to the matching row. `HomeViewModel.AuthorCount` gained a `CanonicalAuthorId` field to support this.

Detail loads lazily on first expand and caches per-author in a dictionary. Structural operations (mark-as-alias, promote, rename) invalidate cached detail for the affected authors so the next expand reflects the updated rollup.

## Surprise

- **Lazy-load-on-expand + cache-per-row needed essentially no fancy plumbing.** I half-braced for needing some state-management framework (a store, observables, something). Turns out the shape is: `HashSet<int> ExpandedAuthorIds`, `Dictionary<int, AuthorDetail> DetailByAuthorId`, `Dictionary<int, AuthorViewMode> ViewModeByAuthorId` — all owned by the VM. ToggleExpand adds/removes from the set; the Dictionary is checked first and populated on miss. Collapse leaves the cache alone so re-expand is instant. Three collections, maybe 30 lines of VM code, done. The lesson-shape is "don't introduce machinery until the simple shape breaks."
- **`MudLink` wrapping a card-like block keeps the click affordance without breaking the visual.** Top-10 author rows on Home now wrap the whole title+count+progress-bar cluster in a `MudLink Href="..."`. No `MudLink` visual noise (Underline=None, Color=Inherit); the row looks identical to before but the entire cluster is tappable. That's the right pattern for "make this list entry a link without making it look like a link" — rather than adding an explicit "View" button per row.
- **`MudListItem T="int" OnClick="..."` swallowed child clicks again (PR 3 déjà vu).** The Books view lists each book as a `MudListItem` that navigates on click. But the nested cover image + text fragments would fire their own event propagation chain and re-enter the handler. The stopPropagation pattern from PR 3 still applies: any clickable MudListItem with interactive children needs `@onclick:stopPropagation="true"` on a wrapper. I didn't hit it this time because the children are non-interactive (`MudImage`, `MudText`), but the rule-of-thumb is worth re-stating: **MudListItem with OnClick + children = think about propagation before testing, not after.**
- **`.FirstOrDefault()` on an `IReadOnlyList<T>` doesn't exist — needs a cast.** Wasted 30 seconds trying to call it in the razor template before remembering `IReadOnlyList<T>` implements `IEnumerable<T>` but the LINQ extension methods come via System.Linq namespace import. Trivial, but the razor compiler's error message on missing extension methods ("does not contain a definition for 'FirstOrDefault'") is less informative than it should be — looks like a bug at first glance.

## Lesson

- **Opt-in rollout held again.** This is the fourth or fifth page to convert from Bootstrap to MudBlazor ("convert as we touch"): Home, Duplicates/MergeBook, the whole Book Detail page arc, and now Authors. No big-bang rewrite, no tracking ticket for "remaining pages to migrate." The pages that aren't MudBlazor yet just haven't needed substantive work. If we'd front-loaded a migration arc we'd have touched pages that didn't need anything else — convert-as-we-touch means conversion rides on already-justified work. Cost-adjusted conversion.
- **"List with per-row drill-down" has a clean default shape.** Three VM collections (expanded set, detail cache, view mode), one lazy loader per detail type, one invalidation hook on any structural write. The page template loops over the list, renders the summary row always, renders the detail when expanded. No virtualization needed at this scale; no state management library needed; no special "select one row at a time" logic (multi-expand is the simpler default and almost always the right UX for browsing). This shape will port cleanly to future pages that need similar affordances — a Publishers page, a Series detail page, etc.
- **Cache invalidation on structural writes is one line per call site.** After each operation that changes what a rollup should return (`MarkAsAliasAsync`, `PromoteToCanonicalAsync`, `RenameAsync`), add `InvalidateDetailsFor(affectedIds)` before `LoadAsync`. The next expand re-queries. You don't need a pub-sub / reactive framework for this — the list of affected ids is small and knowable at the call site. Where it would get complex: any pattern where one author's detail depends on another's *not yet expanded* view. Our alias-rollup is direct (canonical → aliases), so the affected set is tight.
- **Deep-link via query-string survives reload and browser back.** `/authors?expand=<id>` is better than `/authors#author-<id>` because the VM can read it explicitly in `OnInitializedAsync` without listening for hash-change events, AND it survives SSR / client reload without the hash being silently stripped. `[SupplyParameterFromQuery]` on the component parameter gives Blazor the wiring for free. Good default for "pre-open this specific item on load" flows.

## Quotable

The VM for this page has three small state collections (a set of expanded ids, a dictionary of cached details, a dictionary of per-row view modes) and three methods that mutate them (toggle, set view, invalidate on structural change). That's it. No state-management library, no reactive framework, no "selected row + mode" coupling, no optimistic-update plumbing. The temptation to reach for machinery is real — every "list with expandable detail" article online eventually introduces some abstraction. But the simple shape *works* and stays legible across future additions. If we ever need optimistic updates, change-event propagation, or cross-row selection constraints, we'll add them then. Until then, three collections are enough. **The reward for not reaching for machinery is a VM that a new reader can understand end-to-end in 60 seconds.**
