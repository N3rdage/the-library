## Shipped

PR #93 — MudBlazor 9.3 introduced as the component library. Home and MergeBook pages rewritten; every other page untouched (Bootstrap markup still works). Custom `BookTrackerTheme` ships a warm "library" palette — oxblood primary, antique brass secondary, forest tertiary, parchment backgrounds, espresso appbar, sepia text — applied globally via `<MudThemeProvider>` so the pilot pages pick it up without per-component overrides. Dark-mode parked. Navbar stays Bootstrap for now.

## Surprise

- **Default MudBlazor palette looks like a Material Design demo, not a product.** The pilot landed with `<MudThemeProvider />` (no theme passed) and Drew's first reaction was "a bit bright". That's diplomatic — saturated purple-and-teal announces "this is a generic component library" and fights with any product identity that isn't "generic SaaS dashboard". A ~60-line static theme class with nine custom colours totally changed the feel. The lesson is that theming isn't a polish-pass follow-up, it's table stakes for the first PR that uses the library.
- **Bootstrap and MudBlazor coexist on the same page load without a fight.** Both stylesheets ship in `App.razor`. Each page picks a lane — Home is all MudBlazor, Library is all Bootstrap. The `<MudThemeProvider>` + `<MudPopoverProvider>` + `<MudDialogProvider>` + `<MudSnackbarProvider>` quartet in `MainLayout.razor` is harmless on Bootstrap-only pages. Nothing visual leaks between lanes because Bootstrap's `.container`/`.btn` and MudBlazor's `MudContainer`/`MudButton` use disjoint class namespaces and MudBlazor scopes its component styling via its own class hierarchy. The boogeyman "two UI frameworks on one page is chaos" just doesn't materialise when each page commits to one of them.
- **No migration deadline, no cleanup ticket, no big-bang PR.** The explicit rollout strategy is "convert as we touch". Every time a page gets substantive work, it gets rewritten in MudBlazor as part of that PR. Bootstrap pages that never change never convert. Over time, coverage accrues naturally. The appeal: no dedicated migration arc competing with feature work, no "dual-stack cleanup" task that sits in TODO.md for six months, no decision about when to force the last stragglers. The cost: Bootstrap lingers in the codebase essentially forever for low-traffic pages. Drew was comfortable with that trade — cost is "a library we're not actively using still loads", not "a maintenance burden".
- **MudBlazor's `PaletteLight` is the right surface to theme against; per-component style overrides weren't needed.** All nine custom colours plus muted status variants (Success/Info/Warning/Error) land in one record initialiser. No CSS, no Sass recompile, no JS. Typography defaulted to Roboto (MudBlazor ships a Google Fonts link and picks it up automatically).
- **Closing the IDE closes the AI conversation.** Mid-pilot, debugging a Rider build-cache issue, Drew closed the solution to reset Rider's state — which also ended the Claude Code session embedded in that solution, nuking the in-progress conversation. The pilot commit (`54ae908`) had already landed so no code was lost, but the context of *why* the pilot was shaped the way it was, *what* Drew had already reacted to, *which* palette direction we'd been leaning toward — all gone. The theming conversation started from the git state plus the `retros/` folder plus `MEMORY.md`. That re-priming actually worked surprisingly well (the full retro exists because the memory system captured the shape of previous work), but it's a real workflow coupling worth knowing about.

## Lesson

- **Pilot two pages before committing to the whole codebase.** Rewriting one page would have been too small to test claims ("does it play nicely with Bootstrap? what's the theming story? does `IDbContextFactory` still work under MudBlazor's dialog/snackbar providers?"). Rewriting five would have been a migration arc, with all the sunk cost of "well we've started so let's finish". Two pages is the sweet spot: big enough to stress the integration, small enough to back out cheaply if the library turns out to be wrong.
- **"Convert as we touch" is a legitimate migration strategy — for the right kind of codebase.** It requires: (a) each UI framework's classes/namespaces don't collide, (b) no shared component library that needs both variants, (c) a solo dev who can hold "this page is Bootstrap, this page is MudBlazor" in their head without a lint rule enforcing it. All three hold for BookTracker. For a bigger team or a component-heavy design system, the additive-then-cutover pattern from the Work refactor (PR1 adds, PR2 removes) would be the right template. Here, the additive phase never ends on purpose — some pages stay Bootstrap forever because nobody ever needs to touch them.
- **Theme the library in the same PR that introduces it.** Shipping the pilot with default colours was a real miss — Drew saw it and immediately flagged the palette. The fix was cheap (one file, nine hex values), but the sequencing was wrong: "add library" and "theme library" are the same merge, not two. Worth building into the plan shape for any future component-library introduction.
- **Small palette, big impact.** Nine colours + four muted status variants is the entire theme. Not a design system. Not a token library. Not Sass variables. Just `PaletteLight { Primary = "#6B2737", Secondary = "#A67B3A", ... }`. The temptation to over-engineer (spacing tokens, typography scale overrides, component-level style overrides) was strong and explicitly resisted — none of that is needed on day one. If a future page needs something the default spacing doesn't give, address it then.
- **The memory system earns its keep when the session dies unexpectedly.** Losing a conversation is fine *iff* the next conversation can pick up without starting from zero. `MEMORY.md`, the `retros/` corpus, git state, and `CLAUDE.md` between them carry enough signal that "pilot is working, user wants custom theme, incremental rollout, park dark mode" was reconstructable from a one-line user update. The test for "is the memory system actually working" isn't "can future sessions reference it in theory" — it's "can an interrupted session literally resume mid-feature tomorrow". It passed.

## Quotable

"A shelf of leather-bound books should not look like a Material Design demo." The default MudBlazor palette is fine for a generic dashboard — saturated purple, bright teal, that Google-Material-blue info colour. For a library app it's aggressively wrong. Parchment, oxblood, brass, forest, espresso. Nine hex values. That's the whole theme, and it reads as a *product* instead of a *framework demo*. The lesson generalises: every component library's defaults announce the library, not your product. Theming isn't polish, it's the first thing.
