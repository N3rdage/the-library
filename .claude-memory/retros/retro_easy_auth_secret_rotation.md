## Shipped

PR #111 — automated Easy Auth client secret rotation. The previous state was "rotate manually by re-running `deploy.ps1` whenever you remember" with a 2-year fail-safe; that's fine until it isn't.

- **`rotate-easy-auth-secret.ps1`** — standalone script. Generates a fresh password on the `Library-Patrons` app registration, writes it to KV as `AuthClientSecret`, trims old passwords to keep the latest 2 (so the previous-active secret stays valid during App Service's ~24h KV reference refresh window).
- **`.github/workflows/rotate-easy-auth-secret.yml`** — scheduled cron `0 2 1 */6 *` (02:00 UTC, 1st of every 6th month) + `workflow_dispatch` for on-demand. OIDC-auths as the existing `booktracker-ci` SP, installs the Graph + KV PowerShell modules, runs the script.
- **`setup-github-oidc.ps1`** extended to grant the CI SP three narrow permissions needed for the rotation: Key Vault Secrets Officer on the vault, Owner of the `Library-Patrons` app registration, Graph `Application.ReadWrite.OwnedBy`. All three idempotent on re-run.

Secret lifetime stayed at 2 years (fail-safe); rotation cadence is 6 months, so 4 overlapping secrets could exist between cleanups — trimmed to 2 on each run.

## Surprise

- **Granting a permission to another identity is a different privilege level than using the permission yourself.** First run of the updated `setup-github-oidc.ps1` threw a 403 on `New-MgServicePrincipalAppRoleAssignment`. The Graph session had `Application.ReadWrite.All` — enough to *use* that permission, but not enough to *grant* it to another SP. Granting Graph app role assignments needs `AppRoleAssignment.ReadWrite.All` in the caller's session, plus Global Admin or Privileged Role Administrator in the tenant. Two easy conflations to avoid: (a) "I have this permission" vs "I can grant it to others" are distinct. (b) Caller-side Graph scopes vs tenant-level admin roles — both matter, neither alone is enough. Drew unblocked by granting via the Azure Portal UI (which doesn't need the caller-side scope — it runs under the portal's consent flow). The script was patched to (i) request the extra scope on `Connect-MgGraph`, (ii) wrap the grant call in try/catch with an actionable Portal-fallback message.
- **`$ErrorActionPreference = 'Stop'` does not reliably catch Microsoft.Graph SDK errors.** The 403 surfaced, the script kept running, and then it printed the misleading "Granted ..." line. The Graph cmdlets emit error *records* (not terminating exceptions) in at least some failure modes, and the global preference doesn't always intercept those. The fix is belt-and-braces: explicit `-ErrorAction Stop` on each call *plus* a try/catch around anything that can fail for permission reasons. Worth baking into a general "Graph call pattern" — the happy-path silence of these SDKs masks real problems.
- **RBAC-mode Key Vault draws a clean line between control plane and data plane.** Contributor on the RG lets you manage the KV resource (create/delete/rotate keys at the infrastructure layer) but NOT read/write secret data. `Key Vault Secrets Officer` on the KV is a separate role for data-plane write access. Worth knowing because "Contributor covers everything" is a tempting-but-wrong mental model for KV. Same distinction exists in access-policy-mode KVs but less visibly.
- **`KeepLatest = 2` is the Goldilocks number for this rotation cadence.** Less than 2 risks orphaning the previous-active secret before KV ref refresh picks up the new one (App Service could see the secret disappear mid-window). More than 2 accumulates expired passwords cluttering the app reg. Two keeps exactly "the new one + the one App Service is currently using" — minimal, covers the transition, no cruft. The number comes out of the propagation window (~24h) vs rotation cadence (6 months): as long as cleanup runs more frequently than propagation completes, keeping N-1 previous versions is safe.

## Lesson

- **Two-ticket privilege model for automated operations.** When designing an automated identity, think about two questions separately: (1) what operations will it perform (*direct* permissions)? (2) who needs to grant those permissions to it (*meta* permissions)? They often require different roles/scopes. The bootstrap user needs the meta permissions; the automated identity needs only the direct ones. Confusing the two means either "I can't get it set up" (meta missing) or "the automation has too much power" (over-broad direct grants because we conflated them). Applies across Graph, IAM, SQL GRANT WITH GRANT OPTION, Kubernetes RBAC, etc.
- **Defensive Graph-call pattern: `-ErrorAction Stop` + try/catch around every operation that could 403.** Microsoft.Graph SDKs are quirky enough that relying on `$ErrorActionPreference` is insufficient. The cost of adding try/catch per call is small; the cost of a silent-success-then-actually-failed is a misleading log line and a workflow you believe is fine until it isn't.
- **Portal vs SDK as a recovery path.** When script-driven permission operations fail on privilege grounds, the Portal UI is often a lower-privilege fallback because it runs under the signed-in user's interactive consent flow rather than requiring SDK-scope-in-session. Useful to flag in the script's error message rather than leaving the user stuck. Pattern: *"couldn't do X via script because Y; here's how to do X via Portal in 30 seconds"*. The script's failure becomes a pointer instead of a dead-end.
- **Fail-safe overlap for scheduled rotations.** For any "rotate X before it expires" automation: pick a secret lifetime that's well longer than your rotation cadence (2 years vs 6 months here), keep N-1 previous versions alive during propagation, and trim based on count not age so a broken rotation doesn't silently delete the last working version. The combination makes "automation broke once, we have N cycles to notice and fix before users break" a reliable property.

## Quotable

Granting a permission and *granting-a-permission-to-someone* are two different privilege levels. This shows up everywhere — Kubernetes RBAC distinguishes `verbs` on `Roles` from `verbs` on `RoleBindings`; SQL has `GRANT` vs `GRANT … WITH GRANT OPTION`; Linux filesystems separate write permission from permission to change permissions. In Graph specifically, `Application.ReadWrite.All` lets your session act on app registrations; `AppRoleAssignment.ReadWrite.All` lets your session grant app roles to *other* identities. A senior admin has both and never notices the distinction. An automation-building session running as a slightly-less-privileged identity hits the wall immediately. The lesson is specific to the Graph flavour but the mental model — **using vs delegating as two separate privileges** — generalises to every system that has some notion of authority.
