Raw material for the close-out mega-blog on the pragmatic-spine refactor ([[project-backend-refactor-arc]], [[retro-backend-ddd-pilot]], indexed in [[blog-post-backlog]]). These beats live in conversation, not commits — captured now (2026-06-23, after PR4) before compaction. Verify each against the merged code when writing.

**Thesis / the "why" (Drew's framing):** a main reason for the arc was to build an **agent-safe review/update surface** — so any local change is either *contained* or its wider blast radius is *surfaced*. Two complementary halves: **compiler-enforced seams** (project boundaries + record-typed contracts) surface *structural* blast radius at `dotnet build` (the doc's headline goal), and the **behaviour-preserving test adoption** surfaces *behavioural* blast radius the compiler can't see — which is precisely why the "no-op refactor kept finding bugs" (below) happened. Contain-or-surface is the spine the whole blog hangs on.

**Spine — the two most universal lessons:**

- **"The no-op refactor that kept finding bugs."** Behaviour-preserving adoption (lift inline VM writes into aggregates + handlers, keep method signatures so Razor is untouched) repeatedly surfaced *pre-existing latent bugs*, because lifting code forces you to name what the old code actually did. PR1: a notes-wipe data-loss (caught by adopting against the existing tests). PR3: `RemoveWorkFromSeries` left a dangling `SeriesOrderDisplay` — a stranded "4.5" on a work no longer in any series; `ClearSeries` now clears all three fields. PR4: `MarkAsBought` built a Book from a *stale in-memory row* → a duplicate book on a double-click; fixed by loading the item by id and returning null if it's already gone.

- **Solo-user severity calibration.** Per-PR adversarial review found real-but-*unreachable* concurrency findings nearly every time (`SeriesNameGuard` check-then-act TOCTOU; the Publisher/Author/Tag find-or-create race, TD-15). The discipline that emerged: fix the cheap UX ones, **document-and-accept** the unreachable ones *with the rationale written down*. "Correct" can lose to "a single user will never hit this" — but you record why, so future-you (or a second writer) can revisit when the assumption breaks.

**Vignettes:**

- **The aggregate boundary moves under you (PR2).** Drew reframed mid-arc — "is a Work more central than a Book?" — which produced the ref-count lifecycle: a Work is born attached to its first Book and self-orphans at ref-count 0; the orphan-delete is *signalled* by the aggregate (`RemoveFrom` returns "now orphaned") but *executed* by the handler. Became convention C11. The aggregate boundary you start a refactor with is not the one you finish with.

- **A naming convention's hidden Nth-case cost (PR3).** Naming the feature folder after the aggregate works because the plural dodges the type name (`Works/` vs the `Work` type) — until `Series` (singular == plural) makes the namespace collide with the type and forces a `SeriesAggregate` alias. The convention was fine for 2 of 3 instances.

- **Half-rich, half-raw graph assembly (PR4).** The promotion handler used `Work.Create` (rich factory) but hand-rolled `new Edition { Copies = [new Copy] }` — review caught it; `book.AddEdition` is the seam that owns the Edition-has-a-Copy invariant + ISBN normalisation. Lesson: use the factory *everywhere* or the next reader can't tell whether bypassing it was deliberate.

**Meta-layer (a blog in itself):** reviewing your *own agent's* work with N adversarial sub-agent angles on a handed-off diff (subagents can't reliably run git — [[feedback-review-agents-need-diff-file]]), recall-biased single-vote verify, present-findings-and-wait ([[feedback-review-findings-gate]]). Zero reachable bugs across PR1-4 — the value was as much the *documented-accept* decisions as the catches.

**PR6b-1 beats (2026-06-26, read-only display VMs Home/Series/AuthorList → query handlers):**

- **A verbatim relocation draws fire for the legacy code's sins.** The PR6b-1 review's dominant finding (5 of 8 raw candidates) was that `GetAuthorList` rolls the whole join graph up in app memory instead of in SQL — "wrong altitude, and it'll become the house template." But that code was lifted *verbatim* from the old `AuthorListViewModel`; the altitude was preserved on purpose, not authored. The discipline: a behaviour-preserving move inherits the legacy code's smells, and the right response is to **log it (TD-17), not fix it mid-relocation** — fixing would change behaviour and break the "no-op" guarantee the whole arc rests on. Sharpening detail that defused the "house template" worry: the *sibling* handlers in the very same PR (`GetSeriesList`, `GetHomeDashboard`) already push projection/grouping to SQL — only `GetAuthorList` does the in-app thing, because its canonical+alias HashSet rollup is genuinely awkward as one `GROUP BY`. So "the template" was already mixed, and the review over-generalised from one hard case.

- **Three metrics that look like one — where DRY would be a bug.** The refactor co-located three author-count computations that *look* like duplication begging to be unified: Home's top-authors tally (one count per `WorkAuthor` contribution row, so co-authored books count twice), the `/authors` list rollup (distinct works/books/series per canonical author), and the catalog snapshot's `AuthorSnapshot.BookCount` (distinct books, a documented cross-app contract). The original 6b plan literally said "extract a shared author-rollup query." Inspecting them showed they're **three deliberately-different metrics** — collapsing them to one would silently change Home's headline numbers. Kept as three handlers; the duplication is logged (TD-17) as a *latent drift risk* to consolidate behind one query at arc close, not a bug to DRY away now. The blog beat: "duplication" and "the same metric written twice" are not the same thing, and a refactor is exactly when you're most tempted to conflate them.

- **The test-split that falls out of the read-model move.** Each display-VM relocation produced a clean two-test shape: the DB-backed assertions become `Application/<Feature>/Get*HandlerTests` (integration, real SQL), and the VM keeps only thin `IDispatcher`-mock unit tests over the presentation logic that genuinely stayed in the VM (Home's bar-scaling maxima, Series' badge/text helpers + enum parse, AuthorList's free-text/show-aliases filter). The handler owns the data; the VM owns the pixels — and the tests now say so.

**PR6b-2 beats (2026-06-26, AuthorDetail + Publisher admin VMs → queries + the first 6b writes):**

- **Preserve the inconsistency, don't normalise it mid-refactor.** The two admin VMs surfaced the same toast in *different shapes*: AuthorDetail has a two-channel `SuccessMessage` + `ErrorMessage` (rename-clash is an error); PublisherList funnels everything — successes *and* guard refusals ("already exists", "can't delete, still in use") — through one `SuccessMessage`. The tidy instinct is to unify both onto one result shape. Resisted: lifting writes verbatim means the command results mirror what each VM actually did (`AuthorAdminResult` = Success+Error, `PublisherAdminResult` = Changed-flag + single Message). Normalising would have changed observable page behaviour (publisher guard text would move from the success-styled alert to an error-styled one) under cover of a "no-op" refactor. Same discipline as the three-metrics beat: a refactor is where you're most tempted to "fix" a wart that's actually load-bearing UI.

- **The Application.Formatting folder is becoming the gravity well.** `PartialDate`/`PartialDateParser` was the third pure formatter pulled out of `Web.Services` into `Application.Formatting` (after PR6a's `WorkAuthorshipFormatter` + `SeriesOrderParser`) — each move triggered by a read handler needing to project a *display string* (C5: the handler owns the DTO, including its formatted fields). The recurring tax: every display-VM relocation drags its formatters across, because you can't half-move a read — the projection and its formatting must live on the same side of the boundary. Worth naming as a pattern: "the read handler pulls its formatters with it."

**PR6b-3 beats (2026-06-27, BookDetail read + Tag writes → queries/commands + a shared TagResolver):**

- **The relocation that *introduced* a regression — the mirror image of "inherits the legacy sins."** Every prior 6b beat was about a verbatim move *inheriting* a pre-existing smell. 6b-3's review caught the opposite: the move actively *created* a bug. The old `AddTagAsync` built the optimistic tag chip from the resolved `Tag` entity's stored `Name`; the relocated version re-derived the label from the lowercased *input* instead. For a tag matching a pre-existing mixed-case row (resolved via SQL Server's case-insensitive collation), the chip rendered `sci-fi`, then flipped to the stored `Sci-Fi` on the next page load. The failure mode the "no-op refactor" framing misses: when the relocation **re-derives a value the old code read off the entity**, "behaviour-preserving" quietly isn't. Fix — return the resolved entity's value (`TagDetail`, id + stored name) from the command so the real stored name flows back to the VM. Three independent verifier confirmations.

- **A doc comment is a claim; make it true or delete it.** `TagResolver`'s summary declared it "the single owner of tag-name normalisation" — but the trim+lower-case rule was hand-copied in `GetTagSuggestions` and the VM's dedup guard. The review flagged the doc-vs-code gap directly. Cheap fix (extract `TagResolver.Normalize`, route the resolver + the suggestion query through it; the VM's guard switched to `OrdinalIgnoreCase` so it needs no copy) made the claim true. Beat: a "single owner" comment is a falsifiable assertion the next reader will trust — a refactor that writes the claim must also earn it.

- **The test whose assumption the database falsified.** The regression test for the casing fix asserted "no lowercase duplicate row" via `Count(t => t.Name == "sci-fi") == 0` — and *failed on a correct fix*, because SQL Server's case-insensitive collation matches `Sci-Fi` too, so the count was 1. The bug was in the test's mental model (string `==` is case-sensitive), not the code; the real-SQL integration harness caught a gap an InMemory mental model would have sailed past. Collation-agnostic rewrite: assert exactly one row total + its stored casing. (Reinforces the project's standing "InMemory's lax SQL hid real bugs" call — here the lax model was in the *test author's head*.)

- **Simple-by-choice: don't half-close a race.** Drew chose `TagResolver` as a plain find-or-create (no unique-index-catch) to match `AuthorResolver`/`PublisherResolver`, accepting that TD-15's concurrency race stays open for all three. Adding catch+re-query to Tag *only* would have closed it for one of three and created divergence. Consistency over a partial fix — close the race across all three resolvers in one focused pass, or not at all. (Extends the solo-user-severity beat: the *shape* of a deferral matters, not just the decision to defer.)

**PR6b-4 beats (2026-06-27, BookList reads → queries + the TD-6 grouping-dup paydown):**

- **A user heuristic that triggered its own exception.** In the 6b-3 planning Drew stated a standing preference: bias toward fewer/bigger PRs, "unless the scope blows out massively." When 6b-4 was scoped (BookList + Wishlist combined ≈ 25 files, two complex VMs), that very criterion fired and chose **split** — no re-litigation needed. Beat: a well-formed heuristic carries its own escape clause, and naming the clause up front means the later scope call makes itself.

- **Where the risk actually lives: write-surface, not diff size.** The arc's "heaviest" remaining slices (6b-4/6b-5) are the *lowest-risk* per line — every write was already lifted in PR1–5, so these are pure read relocations. A relocated read can't lose data; it can only mis-shape a projection, which the tests catch. The biggest-LOC PR of the arc is mechanically safer than the small write PRs. Risk tracks the write boundary, not the diff.

- **Convention-scan DI makes integration tests follow your relocations for free.** The BookList VM integration tests build a real dispatcher via `AddApplicationLayer()` (convention scan). The instant a new `GetLibrary*` handler *exists*, it's registered and the existing 22 tests exercise it end-to-end through the VM — zero new wiring, no test rewrite. The relocated reads were "covered" the moment the VM dispatched to them. Beat: convention-scan registration means the integration layer of the test pyramid auto-tracks the seam you just moved.

- **A clean test-shape convention earns a heavy-VM exception.** PR6b-1 established the tidy split: DB assertions → `Get*HandlerTests`, VM → thin `IDispatcher` mocks. For the *heavy, incrementally-migrated* VMs (BookDetail, BookList), re-cutting their large green integration suites into mocks would be churn for nothing and would drop the VM-computed-prop coverage (`IsSingleWork`, `TotalCopies`, pagination) — so they stayed integration, now doubling as handler-wiring tests. Beat: a test-shape convention that fits small fully-migrated VMs gets an explicit, recorded exception for the ones migrated piecemeal across many PRs; "don't delete good coverage" beats "be consistent."

**Close-out beats (2026-06-27, consolidation + the whole-arc review — the one place the "no-op" shield drops):**

- **The whole-arc review's job was to review the code the arc itself wrote.** Every prior review judged *verbatim relocations* — and the recurring lesson was "a move inherits the legacy code's sins, log them, don't fix mid-relocation." The close-out is the inversion: `AuthorRollups`/`GenreReads`/`BookCovers` are **new** code that deliberately steps *off* the verbatim discipline to consolidate the logged TD-17 duplication. So the final review is the first one where "I am the author" — the inherits-legacy-sins shield doesn't apply. Drew's sequencing call (pay down arc TD *before* the whole-arc review, not after) is precisely what put this novel code in scope; reviewing after consolidation means the review sees the final state AND the only hand-authored code in the arc. Beat: schedule your most adversarial review for the code you wrote *fresh*, not the code you moved.

- **The ORM wall, hit twice, climbed by the code it was replacing.** The consolidation needed "distinct books per author, rolled to canonical." The obvious SQL — `Select(pair).Distinct().GroupBy(author)` — doesn't translate in EF Core 10. Second attempt, `GroupBy(author).Select(g => g.Select(x=>x.Book).Distinct().Count())` (a literal `GROUP BY ... COUNT(DISTINCT)`) — also walls (`could not be translated`, 29 tests red). The shape that *does* translate is the **per-author correlated subquery**: `a.WorkAuthors.Where(Role==Author).SelectMany(wa=>wa.Work.Books).Select(b=>b.Id).Distinct().Count()`, one row per author, the `COUNT(DISTINCT)` pushed into a correlated subselect. The kicker: that's the exact shape the *old* snapshot `directCounts` already used — the legacy code I was replacing had encoded the workaround years earlier, and the consolidation rediscovered it only after two translation failures. Beat: when the ORM refuses your group-by, the escape is usually to push the aggregate *down* into a per-row correlated subquery — and the code you're deleting may already know that.

- **The first cut shipped the wall's compromise; the review caught it.** Close-1's `AuthorRollups` settled for materialising ~3.6k (author,book) pairs and grouping in app memory — a deliberate retreat *because* the SQL group-by wouldn't translate. That retreat is invisible at 2k books and on /authors (cold, rarely hit), but Home and the mobile snapshot read it on every load. The whole-arc review flagged exactly that bulk transfer as a hot-path scale regression — and only then, prompted, did the correlated-subquery shape surface (the `Author.WorkAuthors` nav made the role filter expressible per-author). Beat: "EF won't translate it" is a real constraint, but it's a reason to escalate the query shape, not to silently accept an O(rows) transfer on a hot path — the wall talks you into the compromise, and it takes a second adversarial pass to notice you took it.

- **The imprecision you write into the comment.** Rolling alias counts onto canonicals by plain SUM means a title credited to *both* King and his Bachman alias counts once per crediting member, not once overall — the cross-member DISTINCT got dropped to climb the same wall. One book in ~2000 (King's Bachman omnibus). Drew: "not worth the drama of keeping the numbers perfect." Same shape as the solo-user-severity beat but about *display correctness*, not concurrency — and the discipline that made it safe was writing the accepted imprecision into the canonical owner's class doc + every consuming comment, so the next reader (and the review) meets a decision, not a discrepancy. A stale comment that still claimed the old "de-duped, counts once" behaviour was itself a review finding (#3) — proof the comment is load-bearing.

**Process beat (this session):** the chat *opened* mid-review — credits ran out during the 6b-2 review the night before. Because the review method materialises the diff to `.debug/pr<N>.diff` first ([[feedback-review-agents-need-diff-file]]), resuming was a byte-identical no-op: the saved diff still matched HEAD exactly, so the review re-ran on the same input with zero reconstruction. A method adopted because *subagents can't reliably run git* paid off for an unplanned reason — *interruption recovery*. Durable intermediate artifacts make long agent workflows restartable across a hard stop.
