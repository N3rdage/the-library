# Tech Debt

A running ledger of code-quality / robustness items surfaced during reviews and
dogfooding ‚Äî the stuff that isn't a feature (those live in [`TODO.md`](../TODO.md))
but that we want a record of, so future-us either fixes it deliberately or can
see we **chose** to live with it and why.

Two sections:

- **Open ‚Äî to address** ‚Äî known debt we intend to pay down when it's convenient
  (usually folded into the next PR that touches the area).
- **Accepted ‚Äî deliberately not fixing (for now)** ‚Äî debt we've looked at and
  consciously decided to leave, with the rationale. Revisit only if the stated
  assumption breaks.

Each item notes where it came from so the context is recoverable.

## Accepted ‚Äî deliberately not fixing (for now)

| # | Area | Item | Decision & rationale |
|---|---|---|---|
| TD-A1 | Library | **Author group drill keys on `group.Label` (canonical name), not the canonical id** ‚Äî unlike Genre/Series which key on `group.Key`. The theoretical `"(unknown)"` label fallback would drill to an empty list. | **Accept.** `Author.Name` is unique (enforced) and `CanonicalAuthorId` is an FK, so the `"(unknown)"` fallback is practically unreachable; name-keying is also *consistent* with the existing name-based author filter shown in the Author autocomplete. Drilling by id would require a new author-id flat filter for no real gain. Revisit only if author-name uniqueness is ever relaxed. (Source: 2026-06-12 review finding #4.) |
| TD-A2 | Library | **An unsubmitted search term is committed to the URL by the next non-search filter edit.** Typing in the search box (which waits for Enter) then changing, e.g., Status serialises the typed-but-not-submitted term too. | **Accept.** Behaviour is unchanged from the pre-URL handlers, and the search box visibly shows the term ‚Äî committing what the user can see when they touch another filter is reasonable. Revisit if it surfaces as confusing in dogfooding. (Source: 2026-06-12 review.) |
| TD-A5 | Mobile | **No auto-retry on the catalog refresh** ‚Äî a post-deploy full-snapshot refresh can fail with a transient transport error (`HttpRequestException`) and succeed on a manual re-tap (server side is clean: ~20 s, no exceptions; the request never completes a round-trip, self-heals on retry). | **Accept ‚Äî manual re-tap, by Drew's call (2026-06-15).** Backoff retry wrapped around a call that can hang ~100 s would freeze the app for minutes with no escape, forcing a cancel button to paper over the retry. Manual re-tap keeps the user in control, and the failure window is rare (only the first refresh after a server-version bump forces a full snapshot; routine refreshes are fast deltas). If it recurs, capture the logcat inner exception (`reset` / `premature` / `Timeout`) to pin the mechanism before reconsidering. (Source: 2026-06-15 debugging session, [[retro_series_order_arc]].) |
| TD-A4 | Library | **An old `?group=Collection` bookmark/shared link silently degrades to the Author grouping** (the Series grouping was retired in #53c). `Enum.TryParse` fails on the retired token and falls back to the Author default; the stale `group=Collection` token is also left in the URL (the page only rewrites the query on a `None`-grouping page clamp). | **Accept.** Sole-user app, the grouped Series view was short-lived, and the fallback can't recover *which* series was being viewed (the query carries no series id), so any remap would be a guess. Landing on the Author default is a safe, non-crashing degradation; the stale token renders fine and only matters if re-shared. Regression-guarded by `ApplyQueryParameters_FallsBackToDefaultsForAbsentOrJunk`. Revisit only if a real shared-link complaint surfaces. (Source: 2026-06-15 #53c review, F2.) |
| TD-A3 | Library | **Author-group drill no longer clusters books series-then-standalone.** The old accordion's expand sorted an author's books with series-having members first (by series name, then SeriesOrder), standalones tailing alphabetically. Drilling an Author group now lands on the flat list sorted newest-first (DateAdded). | **Accept for now.** The clustering sort lived only in the accordion's `LoadGroupBooksAsync`, which became unreachable when PR2 switched to drill-down, then was deleted in TD-1 (2026-06-12). Collection drill still sorts by reading order (SeriesOrder). If author-drill clustering is missed, restore it by porting the sort into `LoadBooksAsync` when `SelectedAuthor` is set ‚Äî but that's a feature decision, not debt. |
| TD-19 | Web / covers | **The SkiaSharp cover pipeline drops EXIF orientation.** `SKBitmap.Decode` ignores the JPEG EXIF orientation tag and `CoverImageProcessor` re-encodes from raw pixels (no tag carried forward), so a cover *uploaded straight from a phone camera* with a non-default orientation could render rotated 90/180¬∞. (`WasNormalised` is still true ‚Äî it's a visual-only regression, not a failure.) Surfaced in the 2026-06-29 ImageSharp‚ÜíSkiaSharp migration review (F2). | **Accept (2026-06-29).** The dominant cover source is upstream providers (Open Library / Google / Trove) ‚Äî already-baked thumbnails with no EXIF orientation, so the common path is unaffected; only the manual phone-upload edge is exposed. It's also unclear ImageSharp's default `Image.Load` (no explicit `AutoOrient`) rotated pixels rather than just preserving the tag, so this may not even be a behaviour change. The mobile cover cache (`CatalogCache.ResizeToJpeg`) decodes the same way and feeds off the same upstream snapshot, so it's consistent. **Fix if a sideways cover actually appears:** read `SKCodec.EncodedOrigin` and apply the transform before encode. (Source: 2026-06-29 migration review F2.) |
| TD-18 | Application & Web / read-models | **"Lead author = first Author-role contributor by `Order`" is duplicated across ~9 projection sites** ‚Äî `GetBookDetail`, `GetAuthorDetail`, `GetLibraryBooks`, `WorkSearchService`, `SeriesEditViewModel`, `AIAssistantViewModel`, `BookAddViewModel`, `DuplicateDetectionService` each encode `WorkAuthors.Where(Role == Author).OrderBy(Order)‚Ä¶` independently (some take `.First()` for the lead deep-link/by-line, some `.Select(Name)` for the full ordered author list; the close-2a detail-page projections added two more). This list is indicative, not exhaustive ‚Äî the grep below is the source of truth. Surfaced in the 2026-06-27 close-2a review (F2). | **Accept (2026-06-28).** No live defect ‚Äî every site picks the same lead author today; the risk is drift-only. A single shared selector **can't be inlined into the ~6‚Äì7 SQL-projection sites** (read handlers) without LinqKit expression-splicing ‚Äî EF Core won't translate a custom method/expression call inside a projection, and we've deliberately avoided that dependency (√Ý la no-MediatR). A plain C# helper would serve only the 2 already-materialised sites while the SQL copies stay duplicated; forcing the rest in-memory to share it would regress the close-2b SQL-projection wins (TD-2/TD-17). Not worth a new dependency to prevent a divergence that hasn't happened in a single-user app. **Revisit if** the lead-author rule itself changes (a new pseudo-role to exclude, a different tie-break): grep the filter predicate `Role == AuthorRole.Author` (filter-only on purpose ‚Äî the read handlers wrap `.Where(...)` and `.OrderBy(...)` across separate lines, so a single-line `).OrderBy(` pattern misses `GetBookDetail`/`GetAuthorDetail`/`GetLibraryBooks`; the predicate over-matches a few non-pick sites like `AuthorRollups`/`WorkAuthorshipFormatter`, which is the safe direction) to find every copy and change them together. (Source: 2026-06-27 close-2a review F2; accept call 2026-06-28.) |
