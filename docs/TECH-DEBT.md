# Tech Debt

A running ledger of code-quality / robustness items surfaced during reviews and
dogfooding — the stuff that isn't a feature (those live in [`TODO.md`](../TODO.md))
but that we want a record of, so future-us either fixes it deliberately or can
see we **chose** to live with it and why.

Two sections:

- **Open — to address** — known debt we intend to pay down when it's convenient
  (usually folded into the next PR that touches the area).
- **Accepted — deliberately not fixing (for now)** — debt we've looked at and
  consciously decided to leave, with the rationale. Revisit only if the stated
  assumption breaks.

Each item notes where it came from so the context is recoverable.

## Open — to address

| # | Area | Item | Why it matters | Size |

| TD-7 | Mobile / scale | **Full catalog-snapshot endpoint is ~20 s for 1,209 books on Basic SQL** (measured via App Insights, 2026-06-15). The `CatalogSnapshotService` projects nested works/editions/contributors/isbns — a heavy shape that climbs with the catalogue. The mobile app forces a full snapshot on the first refresh after any server-version bump, so a slow snapshot is exactly when the transient transport failures (see TD-A5) bite. | Toward the 3000+ target the snapshot time grows and the failure window widens. Best handled in a `/scale-audit` pass (gzip the response, `AsSplitQuery`, projection trimming) — distinct surface from TD-2 (the Library-page load). | M |
| TD-8 | Mobile | **`ThemeColors.Get` resolves design tokens by magic string** (`"BrassTextL"`, `"TextMutedD"`, …) — a typo'd or renamed key compiles cleanly and throws `KeyNotFoundException` at page-render time (`OnAppearing`), crashing the tab rather than showing a wrong colour. Surfaced in the 2026-06-16 Bookshelf-arc review (F6). | The typo surface widens as `Colors.xaml` grows past 60 tokens. A generated/typed token class (or `nameof`-backed constants validated against the dictionary) would move the failure from a runtime page-crash to a build error. Low risk today (keys are covered as pages adopt them); revisit if a typo bites. | S |
| TD-9 | Mobile | **The Wishlist tab re-queries the cache and re-fires a cover fetch per row on every tab visit.** Now that `AppShell` keeps one `WishlistPage` instance for the app lifetime, each return to the tab rebuilds all rows and re-issues N best-effort cover HTTP fetches (`WishlistPage.LoadCoverAsync`). Reload-on-appear is *correct* (it's what fixed the Gaps staleness, F1) — the waste is the un-deduped cover refetch. Surfaced in the 2026-06-16 review (F7). | Visible flicker + wasted bandwidth that scales with wishlist size on rapid tab-switching. Fix is cover-fetch dedup/caching (mirror the Book-cover disk cache, or skip refetch when the row's CoverUrl is unchanged), not a load guard. Low priority — wishlists are small. | S |
| TD-10 | Mobile / build | **Two native libs bundled by Mobile NuGets lack 16 KB ELF page alignment** (XA0141): ZXing.Net.MAUI's camera-core (`libimage_processing_util_jni.so`, via `xamarin.androidx.camera.core` 1.1.0) and `SQLitePCLRaw.lib.e_sqlite3.android` 2.1.2 (`libe_sqlite3.so`). Surfaced by the 2026-06-16 motion-PR `--no-incremental` rebuild. | **Android 16 will require 16 KB page sizes** — on 16 KB-page devices the app could fail to load these `.so`s, which back two core features (the inline barcode scanner and the *entire* offline SQLite cache). We don't own the libs; the fix is upstream package bumps to 16 KB-aligned releases. Watch ZXing.Net.MAUI + SQLitePCLRaw for aligned versions and bump before Android 16 enforcement. | M |
| TD-11 | Build / all projects | **No `TreatWarningsAsErrors`** — build warnings pass silently, and incremental builds only recompile changed assemblies so they don't even reliably reappear. The motion PR's net10 obsolete-API `CS0618`s built "green" on incremental builds and only showed on a forced full rebuild. | Real warnings (deprecations, nullable, analyzer hits) slip through unnoticed until they bite. Enabling warnings-as-errors (our own projects at least) + a non-incremental CI build would catch them at the gate. Can't flip today — there's a pre-existing backlog to clear first (Web `RZ10012`/`MUD0002`, Tests `CS0618` MsSqlBuilder, Mobile XA0141 per TD-10). Plan: clean the backlog, then enable `TreatWarningsAsErrors` with a targeted `NoWarn` for unfixable third-party ones (XA0141). See [[feedback_verify_warnings_clean_build]]. | M |
| TD-12 | Mobile | **The Find `SyncChip` shows a healthy `⟳ {age}` even when the last sync failed or never ran.** It reads `Meta.SyncedAt` whenever online, regardless of success; combined with `AppShell`'s silent launch-sync `catch {}`, a failed/expired launch sync (or signed-out-but-online) shows a normal-looking age with no "stale / sign in / sync failed" cue. Surfaced in the 2026-06-16 Bookshelf-arc review (F7). | The user trusts a stale catalogue as current. Real fix pairs with auth-expiry handling in the online-layer arc (#54): model a sync-failed / never-synced (and ideally `IsSyncing`) state and surface it on the chip. Low risk today (sole user, usually a fresh sync); revisit with #54. | S |
| TD-13 | Mobile | **Three presentation helpers are copy-pasted across the redesign's pages.** Star/status formatting (`FormatStatusRating`/`FormatBookMeta` in `ResultPage` + `FindPage` + `AuthorWorksPage`), the non-disk-cached cover fetch (`ResultPage.LoadEditionCoverAsync` ≈ `WishlistPage.LoadCoverAsync`), and `ResultPage.FormatPrettyName` (camelCase→spaced `BookFormat`). Surfaced in the 2026-06-16 review (F8). | A change to rating display / cover-fetch behaviour / format-name spacing must be hand-applied in 2–3 places and will drift. Extract a `StatusRatingFormatter`, a `Covers.LoadFromUrlAsync`, and move pretty-name into the shared formatter. Low risk; fold into the next Mobile PR touching these pages. | TD-15 | Data / robustness | **Find-or-create on lookup tables has a check-then-insert race.** `PublisherResolver.ResolveAsync` does `FirstOrDefaultAsync(name) ?? (new + Add)` — two concurrent requests creating the *same new* name both miss the SELECT and both insert. The new resolver replicates a **pre-existing** pattern also in `EditionFormDialogViewModel`/`BookAddViewModel`/`BulkAddViewModel` (publishers) and the `AuthorResolver` / Tag find-or-create. Surfaced in the 2026-06-22 PR1a (back-end refactor) review. **2026-06-23 (PR4 review):** the `MarkWishlistItemBought` handler added a *third* inline "follow-up" Tag find-or-create, and it has already drifted from `BulkAddViewModel.EnsureFollowUpTagAsync` (the latter guards `!book.Tags.Any(...)` against double-add, the handler doesn't need to but the literal `"follow-up"` + the resolve logic are now in two places). A shared **`TagResolver`** (mirroring `AuthorResolver`/`PublisherResolver`) is the Tag half of the fix below; deferred so both the handler and the Add-flow VMs adopt it together when those VMs migrate. **2026-06-27 (PR6b-3) — Tag *dedup* half closed:** `TagResolver` shipped (the single owner of tag-name normalisation) and all three tag find-or-create sites now route through it — `AddTagToBook` (BookDetail), `MarkWishlistItemBought`, and `BulkAddViewModel.EnsureFollowUpTagAsync`. **Two strands remain, so this stays Open:** (1) the **concurrency race** itself is still unaddressed — `TagResolver` was deliberately built as a plain `FirstOrDefault ?? new+Add` to stay consistent with its siblings (single-user app, race near-impossible); the unique-index-catch + re-query upgrade should land on **all three** resolvers at once. (2) The **Publisher/Series** inline find-or-creates in `BulkAddViewModel`/`BookAddViewModel` are untouched (those Add-flow VMs aren't migrated yet) — they'll adopt `PublisherResolver`/a new `SeriesResolver` when those VMs move off DbContext. | Near-impossible for the single-user app today, but a real correctness hole (duplicate rows, or a unique-index violation) the moment the app takes concurrent writers. The right fix is **one** shared find-or-create with a unique-index-catch + re-query (or DB upsert) applied across Publisher/Author/Tag — not piecemeal per call site. Low priority while single-user; fold into whichever PR first needs concurrency-safe writes. | S |
| TD-14 | Mobile | **Two altitude items from the arc review (F10).** (a) The ambient `SyncChip` (live online/age, bound to `StateChanged`) lives only on the **Find** tab; Wishlist + Gaps use a static "Sync" `ToolbarItem` with no offline/age signal — the offline indicator is missing on 2 of 3 tabs. (b) If `OnDisappearing` doesn't fire on a tab switch the camera stays live in the background (battery/privacy) — **needs an on-device check**; if it leaks, hook `Shell` navigation instead of the page lifecycle. | S |
| TD-18 | Application & Web / read-models | **"Lead author = first Author-role contributor by `Order`" is duplicated across 6+ projection sites.** `GetBookDetail`, `GetAuthorDetail`, `GetLibraryBooks`, plus `WorkSearchService`, `SeriesEditViewModel`, `AIAssistantViewModel` each encode `WorkAuthors.Where(Role == Author).OrderBy(Order).First()` independently. ## Accepted — deliberately not fixing (for now)

| # | Area | Item | Decision & rationale |
|---|---|---|---|
| TD-A1 | Library | **Author group drill keys on `group.Label` (canonical name), not the canonical id** — unlike Genre/Series which key on `group.Key`. The theoretical `"(unknown)"` label fallback would drill to an empty list. | **Accept.** `Author.Name` is unique (enforced) and `CanonicalAuthorId` is an FK, so the `"(unknown)"` fallback is practically unreachable; name-keying is also *consistent* with the existing name-based author filter shown in the Author autocomplete. Drilling by id would require a new author-id flat filter for no real gain. Revisit only if author-name uniqueness is ever relaxed. (Source: 2026-06-12 review finding #4.) |
| TD-A2 | Library | **An unsubmitted search term is committed to the URL by the next non-search filter edit.** Typing in the search box (which waits for Enter) then changing, e.g., Status serialises the typed-but-not-submitted term too. | **Accept.** Behaviour is unchanged from the pre-URL handlers, and the search box visibly shows the term — committing what the user can see when they touch another filter is reasonable. Revisit if it surfaces as confusing in dogfooding. (Source: 2026-06-12 review.) |
| TD-A5 | Mobile | **No auto-retry on the catalog refresh** — a post-deploy full-snapshot refresh can fail with a transient transport error (`HttpRequestException`) and succeed on a manual re-tap (server side is clean: ~20 s, no exceptions; the request never completes a round-trip, self-heals on retry). | **Accept — manual re-tap, by Drew's call (2026-06-15).** Backoff retry wrapped around a call that can hang ~100 s would freeze the app for minutes with no escape, forcing a cancel button to paper over the retry. Manual re-tap keeps the user in control, and the failure window is rare (only the first refresh after a server-version bump forces a full snapshot; routine refreshes are fast deltas). If it recurs, capture the logcat inner exception (`reset` / `premature` / `Timeout`) to pin the mechanism before reconsidering. (Source: 2026-06-15 debugging session, [[retro_series_order_arc]].) |
| TD-A4 | Library | **An old `?group=Collection` bookmark/shared link silently degrades to the Author grouping** (the Series grouping was retired in #53c). `Enum.TryParse` fails on the retired token and falls back to the Author default; the stale `group=Collection` token is also left in the URL (the page only rewrites the query on a `None`-grouping page clamp). | **Accept.** Sole-user app, the grouped Series view was short-lived, and the fallback can't recover *which* series was being viewed (the query carries no series id), so any remap would be a guess. Landing on the Author default is a safe, non-crashing degradation; the stale token renders fine and only matters if re-shared. Regression-guarded by `ApplyQueryParameters_FallsBackToDefaultsForAbsentOrJunk`. Revisit only if a real shared-link complaint surfaces. (Source: 2026-06-15 #53c review, F2.) |
| TD-A3 | Library | **Author-group drill no longer clusters books series-then-standalone.** The old accordion's expand sorted an author's books with series-having members first (by series name, then SeriesOrder), standalones tailing alphabetically. Drilling an Author group now lands on the flat list sorted newest-first (DateAdded). | **Accept for now.** The clustering sort lived only in the accordion's `LoadGroupBooksAsync`, which became unreachable when PR2 switched to drill-down, then was deleted in TD-1 (2026-06-12). Collection drill still sorts by reading order (SeriesOrder). If author-drill clustering is missed, restore it by porting the sort into `LoadBooksAsync` when `SelectedAuthor` is set — but that's a feature decision, not debt. |
